# 需求文档：真实持仓「卖出 / 平仓」记录功能

> 状态：评审稿 v2 · 2026-06-02
> 关联：我的池子 → 我的持仓（`real_holdings` / 交易流水 / 持仓体检）

---

## 0. 一句话需求

> 「我有可能某一天把一只股票卖掉，我需要把这笔卖出记下来，并且以后还能复盘这笔交易到底赚亏多少。」

现在的持仓系统只能记**买入**和**当前浮盈浮亏**，卖出只能**直接删除**。这会导致记录消失、历史断档、已实现盈亏无法统计。本需求要把真实持仓升级成完整的**买入 → 持有 → 加仓 / 减仓 → 卖出 / 清仓 → 复盘**生命周期。

核心设计原则：**成交流水是事实源，当前持仓是根据流水聚合出来的结果。**

---

## 1. 产品定位和边界

本功能属于 **我的池子 → 我的持仓**，只服务用户手动确认后的真实交易记账。

硬边界：

- 「卖出 / 平仓」是**事后记账**，不是系统给出卖出建议。
- AI 推荐、AI 组合方案、今日决策台可以展示“模型建议减仓 / 组合差额”，但不能自动写卖出记录。
- 用户只有在真实交易发生后，才手动记录买入、加仓、部分卖出、全部卖出。
- 卖出不会自动删除 `watchlist` / 自选股；是否继续关注该股票由用户在「自选股配置」里单独维护。
- 删除按钮保留，但语义是“录错了，抹掉错误记录”；正常卖出必须走「卖出 / 平仓」并保留历史。
- 本功能不改 `model_sim_holdings`，也不把 AI 模拟仓当成真实交易。

---

## 2. 为什么要做（痛点）

| 现状 | 问题 |
|------|------|
| 卖掉股票只能点「删除」 | 这条持仓彻底消失，看不到历史、无法复盘 |
| 没有「已实现盈亏」概念 | 分不清「账面浮盈」和「真落袋的钱」 |
| 当前收益只看还拿着的仓 | 卖掉赚的钱不计入，账户真实累计收益看不全 |
| 不支持卖一部分 | 想先卖一半锁利润、剩一半继续拿，做不到 |
| 同股多笔买入按多行显示 | 用户更想看一只股票当前还剩多少、平均成本多少 |

---

## 3. 参考主流 App 的做法

调研了**雪球 / 富途牛牛 / 老虎证券 / 同花顺**，它们的共识做法：

1. **持仓分两块**：
   - **当前持仓**（Current / Open）= 还拿在手里的
   - **交易历史 / 已清仓**（History / Closed）= 已经卖掉的
2. **卖出 ≠ 删除**：卖出会生成一条成交记录，原持仓数量相应减少；卖光后转入历史，不删。
3. **支持部分卖出**：100 股先卖 50，剩 50 继续算浮盈，平均成本不变。
4. **计算已实现盈亏**：每笔卖出落袋赚/亏多少，按笔、按标的、按年份汇总。
5. **成本法**：默认按**加权平均成本**算，不强制 FIFO / LIFO，对个人记账足够清楚。

本需求采用以上模型，并针对当前系统“单人、手动记账、要能复盘对账”的场景裁剪。

## 4. 本次确认的产品决策

| 决策点 | 结论 |
|--------|------|
| 是否支持部分卖出 | 支持。卖一部分 → 剩余继续展示；卖光 → 当前持仓消失，进入交易历史 |
| 卖出是否保留历史 | 保留。卖出生成成交记录，不删除历史 |
| 同股多次买入 | 用户界面合并成一行，按加权平均成本展示 |
| 事实源 | 交易流水是事实源；当前持仓是流水聚合结果，可做缓存但不能成为唯一账本 |
| 成本法 | 默认加权平均成本，不做 FIFO / LIFO 税务批次 |
| 多币种 | 每笔交易按交易日汇率锁定人民币成本 / 收入，后续统计统一用人民币 |
| 手续费 | P0 支持手续费金额和币种，后端统一折算为人民币 |
| 存量兼容 | P0 必须保持 `fetch_all_real_holdings()` 旧字段契约，同时审计会算钱的下游消费者 |
| 收益口径 | P0 / P1 先叫「累计收益」或「收益曲线」；没有现金流水前，不宣称是完整券商净值 |
| watchlist 联动 | 买入可继续沿用“真实持仓同步自选股”的现有机制；卖出不自动移出自选股 |
| 幂等 / 去重 | P0 必须改用 `client_request_id` / `idempotency_key` 防双击和重试；不能再用“内容全等 + 时间窗口”吞掉重复交易 |

## 5. 核心概念（给零基础的说明）

- **成交流水**：每一次买入、加仓、卖出都是一条不可随意覆盖的记录。这是账本事实源。
- **当前持仓**：把同一 `account + market + symbol` 的当前未清仓轮次按加权平均成本聚合后，得到当前剩余股数、剩余成本和浮动盈亏。已清仓历史只留在交易历史里，不污染下一轮持仓。
- **浮动盈亏（未实现）**：还拿着的股票，按今天价格计算账面赚/亏。没卖就还没落袋，会变。
- **已实现盈亏**：已经卖掉的部分，钱真落袋了，不再随股价变化。
- **累计收益**：已实现盈亏 + 当前浮动盈亏。它是当前账户股票交易结果的近似视图。
- **加权平均成本**：同一只股票多次买入时，用总成本除以总股数。

例子：

```text
买入 100 股 @ $10，汇率 7.0，成本 ¥7000
再买 100 股 @ $14，汇率 7.0，成本 ¥9800

总股数 = 200 股
总成本 = ¥16800
人民币平均成本 = ¥84 / 股
原币平均买入价 = ($10 × 100 + $14 × 100) / 200 = $12 / 股
```

以后不管卖多少股，都按当时这只股票的**剩余平均成本**计算已实现盈亏。

多币种账本必须同时保存两条口径，不能互相折算：

- **原币均价**：用原币价格和股数直接加权，用于页面展示、止损线、与当前原币股价比较。
- **RMB 成本**：每笔交易按成交日锁定汇率和手续费折算，用于真实成本、已实现盈亏、未实现盈亏、累计收益。

不要用“RMB 平均成本 ÷ 当前汇率”倒推原币均价。两笔不同锁定汇率的港股 lot 会因此复现 FX 漂移问题。

买入手续费计入成本基：

```text
买入成本RMB = 买入价 × 买入数量 × 买入汇率 + 买入手续费RMB
平均成本RMB/股 = 累计剩余成本RMB / 剩余股数
```

卖出手续费不计入成本基，直接从本笔卖出已实现盈亏里扣减。

## 6. 功能需求

### 6.1 买入 / 加仓

现有「录入持仓」本质上要升级为「记录买入成交」。

如果 `account + market + symbol` 没有未清仓持仓：

- 新建一条买入流水。
- 聚合出一条当前持仓。
- 当前持仓状态为 `open`。

如果 `account + market + symbol` 已有未清仓持仓：

- 识别为加仓。
- 追加一条买入流水。
- 重新计算总股数、剩余成本、平均成本。
- 当前持仓仍显示为一行，增加「已加仓」标识，可展开看每笔买入。

买入 / 加仓字段：

| 字段 | 必填 | 说明 | 默认 / 校验 |
|------|------|------|-------------|
| 账户 | 可选 | 如 `default`、`ira` | 默认 `default`；P0 API 必须支持，P1 页面提供选择或高级输入 |
| 股票代码 | 必填 | 如 `MCD`、`9992.HK` | 非空 |
| 市场 | 必填 | US / HK / CN | 可按代码推断，可手改 |
| 买入价（原币） | 必填 | 按股票本币填 | > 0 |
| 买入数量 | 必填 | 支持小数股，但 A 股 UI 可按 100 股提示 | > 0 |
| 买入日期 | 必填 | 真实成交日期 | 不可晚于今天 |
| 买入汇率 | 自动 | 1 单位本币 = ? 人民币 | 自动取交易日汇率，可手改 |
| 手续费 | 可选 | 佣金等 | ≥ 0 |
| 手续费币种 | 可选 | 默认同股票交易币种 | 后端折算为人民币 |
| 备注 | 可选 | 如“首次建仓”“补仓” | 文本 |

买入 / 加仓计算规则：

```text
买入成交额RMB = 买入价 × 买入数量 × 买入汇率
买入手续费RMB = 手续费 × 手续费汇率
买入成本基RMB = 买入成交额RMB + 买入手续费RMB
```

加权平均成本必须使用**含买入手续费的成本基**。否则同一笔交易在券商账户和系统账本里的成本会长期对不上。

### 6.2 卖出 / 平仓

在「我的持仓」每一行新增 **「卖出 / 平仓」** 按钮。点开后填写卖出表单。

卖出字段：

| 字段 | 必填 | 说明 | 默认 / 校验 |
|------|------|------|-------------|
| 卖出价（原币） | 必填 | 按股票本币填，如美股填美元 | > 0 |
| 卖出数量 | 必填 | 可小于剩余持有数量 | 0 < x ≤ remaining_shares |
| 卖出日期 | 必填 | 真实成交日期 | 默认今天；不可晚于今天；不可早于首笔买入日 |
| 卖出汇率 | 自动 | 1 单位本币 = ? 人民币 | 自动取卖出日汇率，可手改 |
| 手续费 | 可选 | 佣金、印花税等 | ≥ 0 |
| 手续费币种 | 可选 | 默认同股票交易币种 | 后端折算为人民币 |
| 备注 | 可选 | 如“止盈一半”“触发止损后手动卖出” | 文本 |

提交后系统做的事：

1. 生成一条卖出成交记录。
2. 按卖出前的剩余平均成本计算本次卖出对应的成本。
3. 计算本笔已实现盈亏人民币：

```text
卖出收入RMB = 卖出价 × 卖出数量 × 卖出汇率
本次卖出成本RMB = 卖出前平均成本RMB/股 × 卖出数量
手续费RMB = 手续费 × 手续费汇率
已实现盈亏RMB = 卖出收入RMB - 本次卖出成本RMB - 手续费RMB
已实现收益率 = 已实现盈亏RMB / 本次卖出成本RMB
```

4. 更新当前聚合持仓：
   - 剩余数量 > 0：状态变为 `partial`，当前持仓列表继续显示剩余股数、剩余成本、浮动盈亏。
   - 剩余数量 = 0：状态变为 `closed`，从当前持仓列表移除，进入「已卖出 / 交易历史」。

### 6.3 当前持仓列表

当前持仓列表默认只展示 `open / partial` 状态，不展示 `closed`。

展示字段建议：

- 股票名称 / 代码 / 市场
- 剩余股数
- 平均成本价（原币展示 + RMB tooltip）
- 当前价格
- 当前市值 RMB
- 剩余成本 RMB
- 当前浮动盈亏 RMB / %
- 状态标识：已加仓、已减仓
- 操作：加仓、卖出 / 平仓、编辑备注、删除错误记录

注意：

- 浮盈按**剩余股数 × 当前价**和**剩余成本**计算。
- 前端不重复实现盈亏算法；后端返回已计算字段，前端只渲染。

### 6.4 已卖出 / 交易历史

新增折叠区块 **「已卖出 / 交易历史」**，按卖出日期倒序，使用紧凑表格。

每行展示：

```text
股票名称 · 买入均价 → 卖出价 · 卖出 X 股 · 卖出日期 · 已实现盈亏(¥ / %) · 汇率 · 手续费
```

同一只股票多笔卖出，各算一行。点开可看完整买入 / 加仓 / 卖出链路。

### 6.5 顶部收益统计

在「我的持仓」顶部摘要增加 3 个数：

1. **累计已实现盈亏**：所有卖出记录加总。
2. **当前浮动盈亏**：当前未清仓持仓的浮动盈亏。
3. **累计收益 = 已实现 + 浮动**。

可后续扩展：

- 按标的汇总。
- 按年份汇总。
- 按账户汇总。
- 按市场汇总。

### 6.6 收益曲线 / 净值曲线

P1 先增强现有「账户收益曲线」：把已实现盈亏计入，不让卖出赚钱后曲线凭空掉坑。

口径限制：

- 没有现金流水、入金/出金记录前，不把它叫完整“券商净值曲线”。
- 页面文案建议叫「累计收益曲线」或「持仓收益曲线」。
- 如果未来要做严格账户净值，需要新增现金账户流水：入金、出金、买入占用现金、卖出回笼现金、股息、费用。

## 7. 数据模型建议

### 7.1 推荐原则

推荐采用：**交易流水表 = 唯一事实源；当前持仓表 = 聚合结果 / 查询缓存。**

原因：

- 买入、卖出、手续费、汇率都是交易事实，不能只存在当前持仓聚合行里。
- 当前持仓随卖出、加仓变化，适合作为展示和体检输入，但不适合作为唯一历史账本。
- 未来扩展股息、拆股、券商导入、现金账户时，流水模型更自然。

### 7.2 表结构方向

建议 P0 新增统一交易表：

```text
real_holding_trades
  trade_id
  account
  market
  symbol
  name
  side                  buy | sell
  trade_price
  quantity
  trade_date
  executed_at           可选；真实成交时间，用于同日多笔交易排序
  currency
  fx_rate
  fx_as_of
  fx_source
  gross_amount_rmb
  fee_amount
  fee_currency
  fee_fx_rate
  fee_rmb
  client_request_id      前端/调用方生成的幂等键；同一键重复提交返回同一结果
  order_in_day           可选；同一交易日无法提供 executed_at 时的人工顺序
  position_epoch         持仓轮次：从 0 股开始，到再次回到 0 股结束
  status                active | voided
  realized_pnl_rmb      sell 专用派生快照，buy 为空；rebuild 时可重算
  realized_pnl_pct      sell 专用派生快照，buy 为空；rebuild 时可重算
  cost_basis_rmb        sell 专用派生快照：本次卖出对应成本；rebuild 时可重算
  notes
  source                manual
  voided_at
  void_reason
  created_at
  updated_at
```

当前持仓可以有两种实现：

| 方案 | 做法 | 评价 |
|------|------|------|
| 视图 / 查询聚合 | 每次按 `real_holding_trades` 聚合出当前持仓 | 最干净，但查询和体检需要统一 helper |
| 聚合缓存表 | `real_holdings` 存当前聚合行，每次交易后后端重算写回 | 更贴近现有代码，前端改动较小 |

考虑当前代码已有 `real_holdings`、体检和 API，P0 推荐采用**聚合缓存表**：

```text
real_holdings
  id
  account
  market
  symbol
  name
  total_buy_shares
  remaining_shares
  avg_cost_local_per_share  原币平均买入价，不由 RMB 成本倒推
  avg_cost_rmb_per_share
  cost_rmb_locked        剩余部分成本 RMB，兼容旧字段
  remaining_cost_rmb     与 cost_rmb_locked 同值，语义更清晰
  entry_price            兼容旧字段：原币平均买入价
  shares                 兼容旧字段：remaining_shares
  entry_date             兼容旧字段：当前 epoch 首笔买入日
  currency
  position_epoch
  first_entry_date
  last_trade_date
  close_status           open | partial | closed
  notes
  created_at
  updated_at
```

但必须明确：

- `real_holdings` 不是事实源。
- 所有买卖历史必须能从 `real_holding_trades` 还原。
- 后端提供一个 `rebuild_real_holdings_from_trades()`，必要时可从流水重建聚合表。
- `realized_pnl_rmb`、`realized_pnl_pct`、`cost_basis_rmb` 虽然存放在 sell trade 行上，但它们是**卖出时点的派生快照**；权威输入仍是 active trade 流水。发生撤销、乱序补录或 rebuild 时，后端必须按持仓轮次重算这些派生字段。

### 7.3 为什么不优先拆成 buy_records / close_records

拆成 `real_holding_buy_records` 和 `real_holding_close_records` 也能做，但统一 `real_holding_trades` 更利于：

- 同一个查询按时间线展示完整交易历史。
- 同一套字段处理手续费、汇率、备注。
- 后续扩展现金流水或券商成交导入。

如果实现时为了减少改动临时拆表，也要保持概念一致：**买入记录和卖出记录共同组成交易流水事实源**。

### 7.4 迁移、兼容和 holding_id 风险

当前系统已有 `real_holding_review_items.holding_id` 按现有 `real_holdings.id` 引用。近期代码还支持同 symbol 多 lot 独立体检。因此迁移时必须处理旧 id：

- 迁移前备份当前 `real_holdings` 和 `real_holding_review_items`。
- 对现有每一行 `real_holdings` 回填一条 `buy` trade。
- 按 `account + market + symbol` 聚合生成新的当前持仓。
- 建立 `old_holding_id → new_holding_id` 映射表或迁移日志。
- 历史体检记录可以保留旧 `holding_id` 作为历史快照，不强行改写；新体检只引用新聚合持仓 id。
- 页面查看历史体检时，如果旧 id 找不到当前聚合行，要能显示历史 symbol / account，而不是报错。

更严重的是下游读者兼容。当前 `fetch_all_real_holdings()` 已经是多个核心任务的隐性接口，P0 不能只改表结构后指望调用方自然适配。

已知直接消费者：

| 消费者 | 当前用途 | P0 要求 |
|--------|----------|---------|
| `stock_research/jobs/real_holding_review.py` | 持仓体检核心输入 | 只读取 open / partial；使用兼容字段和后端计算字段 |
| `scripts/pipeline/intraday_refresh_holdings.py` | 盘中只刷新真实持仓 symbol | 只应看到未清仓持仓 |
| `scripts/pipeline/trade_delta.py` | 真实持仓 vs AI 组合方案差额 | 金额必须优先用 `cost_rmb_locked / remaining_cost_rmb`，不能再 `entry_price × shares × 静态 fx` |
| `scripts/pipeline/risk_metrics.py` | 按真实持仓计算权重 / 风险 | 金额必须优先用 `cost_rmb_locked / remaining_cost_rmb` |
| `stock_research/jobs/morning_brief.py` | 早报主入口和止损告警 | 止损用原币 `entry_price`，金额汇总用后端 RMB 字段 |
| `stock_research/jobs/weekly_self_review.py` | 周复盘持仓变化 | `shares` 兼容为剩余股数 |
| `stock_research/jobs/junior_stock_watcher.py` | 次新雷达排除 / 标记真实持仓 | 只应读到未清仓 symbol |
| `stock_research/jobs/defense_watcher.py` | 防御层真实持仓判断 | 只应读到未清仓持仓 |
| `stock_research/api/main.py` | 列表、同步自选、净值 / 摘要接口 | 列表兼容旧字段；收益类接口只用后端单一计算源 |
| `scripts/pipeline/build_stock_dashboard_html.py` | Dashboard 渲染真实持仓 | 渲染兼容字段，同时逐步切到显式新字段 |

兼容字段契约：

```text
fetch_all_real_holdings()
  shares                   = remaining_shares
  entry_price              = avg_cost_local_per_share
  entry_date               = 当前 position_epoch 的 first_entry_date
  cost_rmb_locked          = remaining_cost_rmb
  remaining_shares         = remaining_shares
  avg_cost_local_per_share = 原币平均买入价
  avg_cost_rmb_per_share   = remaining_cost_rmb / remaining_shares
  remaining_cost_rmb       = remaining_cost_rmb
```

注意：字段名兼容只能防止调用方直接报错，不能自动保证所有金额都正确。凡是下游自己用 `entry_price × shares × fx` 算 RMB 成本的地方，P0 必须改为优先使用 `cost_rmb_locked / remaining_cost_rmb`。否则 `9992.HK` 这类两笔不同锁定汇率的 lot 仍会算错。

迁移脚本还必须检查近期已有脚本 `scripts/tools/migrate_holdings_split_v2.py` 对 `real_holdings` 执行过清空 / 拆分逻辑。新迁移要有 preflight，确认当前库状态、迁移 marker 和备份，不得与旧脚本重复清表或重复回填。

### 7.5 rebuild 完整算法

为解决“清仓后再买”的成本污染问题，P0 必须引入**持仓轮次**（`position_epoch`）。

定义：

- 对每个 `account + market + symbol` 独立计算轮次。
- 一轮持仓从该标的剩余股数由 `0 → >0` 的第一笔 buy 开始。
- 这一轮中可以有多笔 buy / sell。
- 当剩余股数再次回到 `0`，这一轮结束，状态为 `closed`。
- 下一次 buy 必须开启新的 `position_epoch`，不能和上一轮 closed 历史继续混算平均成本。

权威重建算法：

1. 只读取 `status=active` 的 trade，voided trade 不参与聚合和收益计算。
2. 对每个 `account + market + symbol` 独立回放。
3. 权威排序键：

```text
trade_date ASC,
executed_at ASC NULLS LAST,
order_in_day ASC NULLS LAST,
created_at ASC,
trade_id ASC
```

4. `trade_date` 是最重要排序键。用户乱序补录时，rebuild 必须按真实成交日期回放，不能按录入顺序回放。
5. 同一交易日同时有 buy / sell 时，优先使用 `executed_at`；没有成交时间时使用 `order_in_day`；两者都没有时才退回 `created_at, trade_id`。不能静默按“先买后卖”改写事实。
6. 遇到 buy：
   - 如果当前剩余股数为 0，开启新 `position_epoch`。
   - 将买入数量加入本轮剩余股数。
   - 将 `trade_price × quantity` 加入本轮原币展示成本，不含手续费。
   - 将 `trade_price × quantity × fx_rate + fee_rmb` 加入本轮 RMB 剩余成本。
7. 遇到 sell：
   - 必须归属当前 open epoch。
   - 卖出数量不得超过当前剩余股数；如果按权威排序回放出现负数，rebuild 必须失败并报告具体冲突 trade，不能产出错误聚合缓存。
   - 本次卖出 RMB 成本 = 卖出前 `avg_cost_rmb_per_share × quantity`。
   - 本次卖出原币展示成本 = 卖出前 `avg_cost_local_per_share × quantity`。
   - 计算并回写 sell trade 的 `cost_basis_rmb`、`realized_pnl_rmb`、`realized_pnl_pct` 派生快照。
   - 从本轮剩余股数、原币展示成本、RMB 剩余成本中扣除本次卖出对应部分。
8. 如果卖出后剩余股数为 0，将本轮标记为 closed。
9. `real_holdings` 默认只缓存每个标的最新 open / partial epoch；closed epoch 只在交易历史中展示。

计算字段：

```text
avg_cost_local_per_share = remaining_local_trade_price_basis / remaining_shares
avg_cost_rmb_per_share   = remaining_cost_rmb / remaining_shares
entry_price              = avg_cost_local_per_share   # 兼容旧字段
cost_rmb_locked          = remaining_cost_rmb         # 兼容旧字段
```

已实现 / 未实现汇率口径：

- 已实现盈亏用成交日锁定汇率：卖出收入按 sell trade 的 `fx_rate` 折 RMB，卖出成本按本轮已锁定 RMB 平均成本扣减。
- 当前未实现盈亏用当前市价和当前汇率盯市：`当前市值RMB = current_price × remaining_shares × current_fx_rate`，再减 `remaining_cost_rmb`。
- `pnl-summary` 的 `total_pnl_rmb = realized_pnl_rmb + unrealized_pnl_rmb`，但两者汇率口径不同，不能共用一个 fx。

例子：

```text
Epoch 1:
  buy 100 @ $10
  sell 100 @ $12
  remaining = 0，Epoch 1 closed

Epoch 2:
  buy 20 @ $13
  当前平均成本只能来自 Epoch 2，不能被 Epoch 1 的 $10 成本污染
```

### 7.6 迁移执行 runbook

这不是离线新系统初始化，而是在一个已有常驻 API、早报和后台任务的生产库里改账本。迁移步骤必须具体到操作顺序：

1. 宣布维护窗口，暂停页面真实持仓写入入口。
2. 停止 launchd 常驻 API，释放 `stock_history_v2.duckdb` 写连接。
3. 停止或跳过可能读写真实持仓的后台任务：盘中刷新、持仓体检、早报触发链路。
4. 备份 DuckDB、`real_holdings`、`real_holding_review_items`、相关 dashboard / latest 输出。
5. 迁移 preflight：
   - 检查 `real_holding_trades` 是否已存在有效数据。
   - 检查迁移 marker，防止重复回填。
   - 检查 `scripts/tools/migrate_holdings_split_v2.py` 是否已对当前库做过拆分 / 清空。
   - 输出将回填多少旧 lot、将聚合成多少当前持仓。
6. 执行迁移：回填 buy trade、rebuild 聚合缓存、写迁移日志。
7. 校验：
   - 当前未清仓总股数、RMB 剩余成本与迁移前一致。
   - `fetch_all_real_holdings()` 兼容字段完整。
   - 持仓体检、早报、风险、调仓 diff 的最小回归通过。
8. 重启 launchd API。
9. 手动打开 dashboard / API 列表确认真实持仓只读展示正常。
10. 恢复后台任务。

迁移期间如果任一步失败，不允许带着半迁移状态继续启动 API。应回滚到备份库或修复后重新执行 idempotent migration。

## 8. 后端 API 和计算规则

### 8.1 API

P0 建议新增：

```text
POST /api/real-holdings/{id}/add      # 接收 client_request_id / idempotency_key
POST /api/real-holdings/{id}/close    # 接收 client_request_id / idempotency_key
GET  /api/real-holdings/{id}/records
GET  /api/real-holdings/trade-history
GET  /api/real-holdings/pnl-summary
POST /api/real-holdings/rebuild-from-trades
POST /api/real-holdings/trades/{trade_id}/void
```

现有接口调整：

- `GET /api/real-holdings` 默认只返回 `open / partial`。
- `POST /api/real-holdings` 继续可用，但语义改为“记录买入成交”；如果同股已有未清仓持仓，后端按加仓处理，并同样接收 `client_request_id` / `idempotency_key`。
- `DELETE /api/real-holdings/{id}` 只用于删除没有交易历史的错误聚合行；正常录错交易走 void trade，不硬删事实源。
- `POST /api/real-holdings/trades/{trade_id}/void` 用于 P0 纠错，只支持撤销该 `account + market + symbol + position_epoch` 下最新一笔 active trade。撤销后后端必须 rebuild 聚合持仓和派生 PnL。

撤销规则：

- P0 只支持撤销**最近一笔 active trade**，不支持任意中间交易修改。
- 撤销不是物理删除，而是将 trade 标记为 `status=voided`，写入 `voided_at`、`void_reason`。
- 撤销后必须从 active trade 流水重建对应标的的 `real_holdings` 聚合缓存。
- 如果要撤销的不是该轮次最新 active trade，返回 409，并提示“P0 仅支持撤销最近一笔交易”。
- 撤销 sell trade 后，已实现盈亏摘要和交易历史必须同步更新。
- 撤销 buy trade 后，剩余股数、平均成本、close_status 必须同步更新；如果撤销导致该 epoch 不再有 active trade，当前持仓应消失。

幂等规则：

- 同一个 `client_request_id` 重复提交，必须返回同一条 trade / 同一聚合结果，不重复记账。
- 不同 `client_request_id` 即使 `account + market + symbol + price + quantity + trade_date` 完全相同，也必须视为两笔合法成交。
- 不能用“内容全等 + 10 分钟时间窗口”作为买入 / 加仓去重依据；真实交易中同价同量分两单成交是合法场景。
- 如果调用方没有传幂等键，前端可以禁用按钮防双击，但后端不能静默吞掉内容相同的第二笔交易。

### 8.2 后端单一计算源

平均成本、剩余成本、已实现盈亏、累计收益必须全部在后端计算一次。

前端禁止自己重新计算这些核心字段，只能渲染后端返回值。原因：

- 多币种汇率容易出现双引擎漂移。
- 手续费币种和折算规则必须统一。
- 后续如果加入现金流水，不希望前端再改一遍财务逻辑。

后端对外至少提供一个统一函数：

```text
fetch_pnl_summary()
```

命名不要只叫 `fetch_realized_pnl_summary()`，因为 API `/pnl-summary` 要返回：

```text
realized_pnl_rmb
unrealized_pnl_rmb
total_pnl_rmb
as_of
realized_since
```

其中：

- `realized_pnl_rmb` 从 active sell trade 的派生快照汇总，快照由 rebuild 重算。
- `unrealized_pnl_rmb` 从当前 open / partial 持仓、当前价格、当前汇率计算。
- `total_pnl_rmb = realized_pnl_rmb + unrealized_pnl_rmb`。

### 8.3 校验规则

买入校验：

- `quantity > 0`
- `trade_price > 0`
- `trade_date <= today`
- `fx_rate > 0`
- `fee_amount >= 0`
- `client_request_id` 若提供，必须在同账户交易写入路径内幂等唯一
- 买入手续费必须计入成本基

卖出校验：

- `quantity > 0`
- `quantity <= remaining_shares`
- `trade_price > 0`
- `trade_date <= today`
- `trade_date >= first_entry_date`
- `fx_rate > 0`
- `fee_amount >= 0`
- 同一账户、市场、股票不能因为乱序交易导致任一时点剩余股数为负。
- `client_request_id` 若提供，必须在同账户交易写入路径内幂等唯一。

## 10. 持仓体检和收益曲线影响

### 10.1 持仓体检

- `closed` 持仓不进入每日体检。
- `partial` 持仓按剩余股数、剩余成本、当前市值计算。
- 体检只评价用户真实当前持仓，不评价已清仓历史。
- 历史交易可作为复盘材料，但不能自动改变 AI 推荐策略版本。

### 10.2 行情刷新

`intraday_refresh_holdings.py` 只刷新 `open / partial` 当前持仓里的 symbol。

### 10.3 收益曲线

P1 先做：

```text
累计收益 = 累计已实现盈亏 + 当前未实现盈亏
```

P2 如果要做严格账户净值，再新增现金流水。

上线当天的历史曲线要明确处理断档：

- 旧每日快照没有“截至当日累计已实现盈亏”字段，不能伪造历史已实现收益。
- P1 默认从上线日开始设置 `realized_pnl_baseline_rmb = 0`。
- 页面文案标注：`已实现盈亏自 YYYY-MM-DD 起统计`。
- 如果未来要补历史，需要从完整交易流水或券商成交导入回放，不能用当前持仓倒推。

## 14. 评审批注

### 14.1 当前 CRUD 去重测试的定位

现有 `tests/test_real_holdings_crud.py` 里的 `test_concurrent_duplicate_create_cleans_inserted_lot` 可以保留为当前模型的回归保护，但它不能被理解为“已经证明真实并发安全”。

原因：

- 这个测试通过 monkeypatch 私有函数 `_recent_duplicate_real_holding_id`，模拟“插入前检查漏掉、插入后兜底检查命中”的分支。
- 它验证的是兜底删除逻辑能不能把刚插入的重复 lot 清掉。
- 它不是两个 API 请求、两个 DuckDB connection 同时写入的真并发测试。
- 真正的生产竞态取决于 DuckDB 事务可见性、连接提交时机、锁行为，以及第二个请求插入后检查能不能看到第一个请求尚未提交或刚提交的行。

因此，当前测试给的是**逻辑分支信心**，不是**跨连接并发证明**。

如果继续保留该测试，建议：

- 改名，避免使用 `concurrent` 这种容易误导的词。
- 增加注释说明它是“兜底分支测试”，不是生产级并发测试。
- 断言 `dedupe_deleted_id`，确保测试确实走到了“插入后删除新 lot”的分支，而不是只靠表里剩 1 行间接推断。

### 14.2 v2 不应继承旧的内容去重语义

旧模型里的 10 分钟窗口去重逻辑，本意是防双击 / 网络重试。但它的判重条件是交易内容全等：

```text
account + market + symbol + price + quantity + date
```

在 v2 交易流水模型里，这会和“同股多次买入 / 加仓是合法交易”发生冲突。

典型误伤：

```text
10 分钟内真实成交两笔 MCD：
第一笔：100 股 @ $10
第二笔：100 股 @ $10
```

这在真实券商成交里完全可能发生，但旧去重会把第二笔静默吞掉，导致少记一笔加仓。

因此 v2 必须改成：

- 防双击 / 重试靠 `client_request_id` / `idempotency_key`。
- 同一幂等键重复提交才视为同一请求。
- 不同幂等键即使交易内容完全相同，也必须记为两笔合法交易。

### 14.3 v2 自动化测试方向

v2 落地后，不要继续围绕 `insert_real_holding_result()` 和 `_recent_duplicate_real_holding_id()` 加测试。那是当前 `real_holdings` 直插模型的白盒测试，生命周期很短。

新测试应转向：

- `real_holding_trades` 写入行为。
- 幂等键重复提交只生成 1 条 trade。
- 不同幂等键 + 内容完全相同生成 2 条 trade。
- 双连接 / 双请求并发写入同一幂等键，不重复记账。
- 双连接 / 双请求并发写入不同幂等键，不互相吞单。
- 清仓后再买开启新 `position_epoch`，不污染新一轮平均成本。
- 撤销最近一笔 trade 后，聚合持仓和派生 PnL 可正确 rebuild。
- 从交易流水重建 `real_holdings` 聚合缓存后结果一致。

### 14.4 十个硬问题

1. **账本事实源问题**  
   `real_holding_trades` 必须是事实源，`real_holdings` 只能是当前聚合缓存。否则卖出、撤销、迁移、收益曲线都会继续依赖易变的聚合行。

2. **下游读者爆炸半径问题**  
   `fetch_all_real_holdings()` 已经被早报、体检、盘中刷新、风险、调仓 diff、周复盘、API、dashboard 等路径读取。P0 必须保持旧字段兼容，并审计金额计算调用方。

3. **rebuild 排序回放问题**  
   加权平均成本依赖交易顺序。rebuild 必须按 `trade_date, executed_at/order_in_day, created_at, trade_id` 回放，不能按录入顺序随便聚合。

4. **持仓轮次问题**  
   清仓后再买必须开启新的 `position_epoch`。否则 rebuild 虽然可能算出正确净股数，但会把上一轮 closed 历史的成本混进新一轮平均成本。

5. **录错交易的纠错路径问题**  
   P0 必须至少支持撤销最近一笔 active trade，并从流水 rebuild 聚合缓存。否则用户填错价格 / 数量后只能硬删聚合行，破坏“交易流水是事实源”的设计。

6. **holding_id 迁移问题**  
   当前 `real_holding_review_items.holding_id` 已经引用旧 `real_holdings.id`。迁移时不能粗暴合并并改写历史体检记录。历史可保留旧 id，新体检引用新聚合 id，页面需要兼容老 id 找不到当前持仓的情况。

7. **幂等和合法加仓冲突问题**  
   防双击不能靠“内容全等 + 时间窗口”。v2 必须使用请求幂等键，否则会误吞真实的同价同量加仓。

8. **原币 / RMB 双口径问题**  
   原币均价要按原币成交价加权；RMB 成本要按每笔锁定汇率和手续费计算。不能用当前汇率在两者之间倒推。

9. **收益曲线口径问题**  
   P1 可以做“累计收益曲线 = 已实现 + 未实现”，但没有现金流水前不能宣称完整券商净值。严格净值必须等 P2 加现金流水后再做。

10. **迁移运维问题**  
   DuckDB 单写者和 launchd 常驻 API 会让迁移脚本拿不到独占写入。迁移 runbook 必须明确停 API、备份、迁移、校验、重启。

### 14.5 并行会话和后台任务提醒

真实持仓会被多个入口触发：

- 用户在两个浏览器 tab 里同时点保存。
- API 请求重试。
- 手机 / 外部入口未来可能提交交易记录。
- `intraday_refresh_holdings.py` 和 `real_holding_review` 可能在读当前持仓。
- 迁移脚本可能在重建 `real_holdings` 聚合缓存。

因此 P0 需要明确并发策略：

- 交易写入必须在后端事务内完成：写 trade、重算聚合持仓、返回结果要么全部成功，要么全部失败。
- 迁移 / rebuild 期间应停止 API 或进入维护模式，避免前端写入。
- 体检和行情刷新只读取 `open / partial` 聚合持仓；如果遇到 rebuild 中间态，应该 fail-soft 或等待重试，不能写出错误快照。
- 幂等键是跨会话防重的核心，不是前端按钮禁用的补充品。
