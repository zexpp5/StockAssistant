> 一句话目标：**核心快线保准点交付，增强异步保数据完整，状态机异常路径自愈。**

注意：本文是**流水线执行分层**（核心/增强/夜班，按"挡不挡交付"切），

## 一、触发事故（2026-06-08）

- 昨晚 21:08 的夜班 `--research` **挂死在第一步 0c**（`refresh_system_universe`），持有 PID 锁。
- 6h 看门狗本该 03:08 强杀，但**笔记本合盖休眠把看门狗的 `sleep` 倒计时和卡死进程一起冻结**了，按"清醒时长"算没到 6h，没触发。
- 锁不释放 → 今早 morning 刷新被跳过 → 用户开机时"数据没拉"。
- 手动清锁重跑后，**早班跑了 67 分钟**；且 `24c` 周一 SEC 扫描超 180s 被杀 → `FAILED_STEPS` 非空 → **飞书简报被拦未推**（简报内容其实齐全）。

## 五、改造规格

### 5.1 `run_step` v2 —— 统一原语（主线 + 子进程共用）

```
run_step <label> <cmd> [--critical|--enhance] [--timeout N]
  ├─ bash 自实现超时（系统无 gtimeout）：后台跑 cmd，看门狗超 N 秒 TERM → 10s 后 KILL
  ├─ --enhance 步 失败/超时 → 记 DEGRADED，继续（永不 return 非零阻断）
  ├─ --critical 步 失败 → 记 FAIL，可阻断
  └─ 退出码语义：只有 CRITICAL FAIL 才让 pipeline 整体 FAIL
```
- 默认超时：核心步宽松（如 600s，必须完成）；轻量事件 90s；增强步各自设（SEC 类 300–600s）。
- 这是 R1 的根治：任一步挂死拖不垮全链，也不再依赖被休眠冻结的全局看门狗（R2 顺带缓解）。

### 5.2 验收闸门改造（R3 根治）

- `production_acceptance_check.py`：`failed = [s for s in steps if s.status=="FAIL" and s.critical]`——DEGRADED 与增强类 FAIL 一律不拦。
- `daily_refresh.sh:672` 推送门：`if 无 CRITICAL FAIL` 才算可推（**不再 `FAILED_STEPS -eq 0`**）。
- 27 对增强类输入（23d6/23d7 US shadow/preflight、主题证据等）：**读最近可用 + 标新鲜度；缺失/过期只 WARN/DEGRADED，不阻断早班交付**（← 修正2，与 F-Score 同一降级哲学）。

### 5.3 F-Score 策略（R4 根治）

- **主线**：不全量重算；picks 经 `LEFT JOIN factor_metadata` 读缓存 f_score；缺失 = `NULL` = 少一个因子（优雅降级，不崩）。可选 `--only-missing` 补新进池标的（平时 0~几只、秒级）。
- **增强子进程**：`compute_piotroski_v2.py` 全量重算，刷新 `computed_at`。
- **新鲜度**：dashboard 标「F-Score 截至 `<computed_at>`」。`factor_metadata` 已有 `computed_at` 字段，现成。
- 只有"完全无缓存"才降级，绝不卡主线。

### 5.4 增强子进程规格（独立生命周期）

| 要求 | 落地机制 |
|---|---|
| 独立 PID | 自己的 `.enhancement_refresh.pid`，**与主线 PID 完全解耦** |
| 启动 reconcile | 发现死 PID 的旧锁 → 清；发现 stale RUNNING 状态 → 标 INTERRUPTED |
| 每步超时 → DEGRADED 继续 | 复用 `run_step v2 --enhance --timeout` |
| 总看门狗 | 子进程自带总超时（如 45min），防再挂 11 小时（R1/R2 在子进程侧的复发防护） |
| 终态写入 | 跑完写 `enhancement_status.json`，异常终止 trap 写 INTERRUPTED |
| 不改已发结论 | **只写增强类 JSON/DuckDB + 写状态 + 轻量重建 dashboard 增强块**；不碰 picks/plan/brief |
| fork 时机 | 主线 **28 push 之后** `nohup … &`；此时 DuckDB 已无主线争用，增强独占写（唯一并发写者只剩每小时持仓刷新，持锁 <30s → 增强步 DuckDB 写加 3 次 retry 兜住） |

### 5.5 新鲜度表面（必须做，否则用户分不清新旧）

- 子进程写 `data/latest/enhancement_status.json`：每块 `{section, last_refresh, status: fresh|stale|degraded}`。
- dashboard 对应板块标：「SEC 数据截至 06-08 10:4x」/「⏳ 增强刷新中」/「⚠️ 今日未刷新（用昨晚缓存）」。
- 这是**新增前端面**，不是改文案，工作量计入。
- 顺带：股票池「数据集」筛选 chip `IPO` → `IPO（暂停）`（与 IPO/次新降权一致）。

### 5.6 状态机自愈（R5 根治）

- `trap` 退出时：若未写过终态，补写 `INTERRUPTED`（覆盖 TERM/正常退出；kill -9 由 reconcile 兜）。
- 启动时 reconcile：发现 RUNNING 状态文件但对应 PID 已死 → 标 INTERRUPTED。
- **立即执行项**：当前 stale 的 `pipeline_status_research.json`（status=RUNNING、updated 06-07T21:08、steps 空）当场改成 INTERRUPTED。

## 七、验证口径（交付前自测，逐条 ✅/❌）

1. `--morning` 实测：核心线 **≤10min 出简报**（对比今天 67min）。
2. **注入失败**：手动让某增强步报错 → 飞书**照常推**、27 不 FAIL。
3. 增强子进程独立 PID；跑完写 `enhancement_status.json` + dashboard 增强块刷新。
4. **DuckDB 无撞锁**：主线写完才 fork，日志无 `Conflicting lock`。
5. **状态自愈**：kill 子进程 → 下次启动 reconcile 标 INTERRUPTED，不再卡 RUNNING。
6. 当前 stale `pipeline_status_research.json` 已清成 INTERRUPTED。

## 八、风险 / 边界

- **fork 时机**必须在 28 push 之后，否则增强写库与主线撞 DuckDB 单写锁。
- **验收 / dashboard 对增强输入统一走「读缓存 + 标新鲜度 + 缺则降级」**，任何一处漏了就会回到"周边拌倒交付"。
- **子进程总看门狗不可省**，否则今天的挂死换个进程重演。
- 完整性反而比现状更好：增强**每天早上并行刷新一遍**（不只夜班），夜班再兜一次；今天那种"夜班挂了一天没数据"的场景不再发生。

## 实施纪律（沿用既有约定）
- 每改完一个独立问题就 commit，只 stage 相关文件。

- 不偷改打分公式；F-Score 走的是"挪计算时机"不是"改算法"。
