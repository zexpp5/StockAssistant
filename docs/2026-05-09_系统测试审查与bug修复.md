## 二、发现的 Bug 与修复

### 🔴 严重（关键管线断裂）

#### B1 · 双 DuckDB 路径分裂

**症状**：`stock_db.py` 写到 `StockAssistant/stock_history.duckdb`，`stock_research/config.py` 默认 `BASE_DIR=~/.hermes/scripts/` 写到另一份 DB。`daily_refresh.sh` 同一次运行同时写两份不同步的 DB。

```
.hermes/scripts/stock_history.duckdb : picks=44 prices=74 reviews=36
StockAssistant/stock_history.duckdb  : picks=12 prices=77 reviews=0
```

**修复**：[stock_research/config.py:10-14](../stock_research/config.py#L10-L14) 默认 `BASE_DIR` 改为 stock_research 包父目录（项目根），与 stock_db.py 一致。

```python
_DEFAULT_BASE = Path(__file__).resolve().parent.parent
BASE_DIR = Path(os.environ.get("STOCK_RESEARCH_BASE", str(_DEFAULT_BASE)))
```

---

#### B2 · reverse_validate v2-v6 全部失败 + factor_scores 命名分裂

**症状**：v3/v4/v5/v6 直接报 `FileNotFoundError: factor_scores.json`，v2 报 `early_signals.json` 不存在。根因——命名不一致：

| 生产者 | 文件名 | Schema |
|---|---|---|
| `factor_model.py` | `factor_scores.json` | `{"results": [...]}` |
| `early_signals.py` | `early_signals.json` | `{"results": [...]}` |
| `daily_picks_v5.py` 缓存 | `factor_scores_today.json` | `{"date":..., "factors":[...], "signals":[...]}` |

`daily_refresh.sh` 只跑 `daily_picks_v5.py`（产合并缓存），不单独跑 factor_model.py / early_signals.py，所以独立 json 永远缺失。

**修复**：

1. 在 [reverse_validate.py:51-71](../reverse_validate.py#L51-L71) 加共享 loader `load_factors_and_signals()`：优先读 `factor_scores_today.json` → 拆成兼容旧 schema 返回；缺失时回退独立 json。
2. v3/v4/v5/v6 全部改用此共享 loader（删除各自的 `json.load(...)` 直接调用）。

修复后 v3-v6 输出：

```
v3: 命中 2  误报 0  漏报 5  正确避开 5
v4: 命中 4  误报 0  漏报 3  正确避开 5  召回 57% 准确 100%
v5: 命中 4  误报 0  漏报 3  正确避开 5  召回 57% 准确 100%
v6: 命中 4  误报 0  漏报 3  正确避开 5  召回 57% 准确 100%
```

## 三、数据迁移记录

### 双 DB 合并

修 B1 后改 `BASE_DIR` 默认值，老 DB 数据被孤立。用 DuckDB `ATTACH` + `INSERT ... ON CONFLICT DO UPDATE` 合并：

```python
new.execute("ATTACH '~/.hermes/scripts/stock_history.duckdb' AS old (READ_ONLY)")
new.execute(f"INSERT INTO picks SELECT * FROM old.picks ON CONFLICT (pick_date, code) DO UPDATE SET ...")
# 同样合并 prices / reviews
```

合并结果：

| 表 | 老 → 合并后 | 增量 |

| picks | 12 → 47 | +35 |
| prices | 77 → 77 | +0（已重叠） |
| reviews | 0 → 36 | +36 |

## 四、系统行为观察

> 这一节不是 bug，是审查中发现的运营层语义问题，记录给后续决策参考。

### O1 · 历史 picks 不会自动下架（运营风险）

**现象**：[daily_picks.py:348-361](../daily_picks.py#L348) 的 `fetch_existing_picks_today` 只查"入选日期=今天"的现存记录跳过去重，昨天及之前的入选记录完全不动。

**后果**：

- 5/9 某股 85 分入选 ⭐⭐⭐，飞书表写一条 `跟踪状态=🟢 在选中`
- 5/10 该股暴跌得分掉到 30，daily_picks 不会再写它，**但 5/9 那条记录仍标 🟢**
- 飞书表持续累积历史 picks，看起来像一直在推荐
- 当前飞书 picks 表有 78 行，已显著超过单日推荐量

### O2 · daily_picks_v5（v6 体系）当天去重失效

[daily_picks_v5.py:179-180](../daily_picks_v5.py#L179):

```python
# v6 学术因子模型与 v1 旧体系并存，不互相跳过 —— 用「入选评分」字段区分
exclude_codes = set()
```

注释说设计意图是 v1/v6 并存（用「入选评分」字段区分），但实际代码是**完全空 dedup**——v6 同日连续跑两次会写两条相同记录。

### O3 · 主体是固定策略，不是自适应学习

| 层 | 是否动态 |
|---|---|
| 价格 / PE / PEG / 涨幅（每日 yfinance 刷新） | 动态 ✅ |
| 季度财报因子（Piotroski / PEAD） | 动态 ✅ |
| 13F 持仓变动（每季度 SEC） | enrich 元数据，**不影响选股** |
| Regime 过滤（VIX / 200MA） | 只 `realtime_defense` 用，**选股不读** |
| 因子 IC 校准（`calibrate_pick_weights` → `data/factor_weights.json`） | [daily_picks.py:425-431](../daily_picks.py#L425) 自己注释"仅作展示，不直接改 score_xxx" — **没接通** |
| 打分权重 v1 (35/25/25/15) | 硬编码 ❌ |
| 入选门槛 v1 (75/60/50) | 硬编码 ❌ |
| 学术因子 v6（4 因子等权 + tertile） | 硬编码 ❌ |
| `THEME_MAPPING` 88 条 | 硬编码 ❌ |
| watchlist 78 只 | 飞书表手工维护 ❌ |
| Markowitz 约束（max 15% / min 2% / cash 5% / lookback 252d） | 硬编码 ❌ |

**结论**：StockAssistant 是**静态规则 + 动态数据**系统。即使一年不改代码、watchlist 不动，每天推荐名单仍会变化（因为价格涨跌让分数排名重排），但**永远是同一套权重 + 同一个池在筛选**。要"自适应"需要：

1. 把 `factor_weights.json` 的 IC 校准结果接到 `score_*()` 函数（当前断）
2. 加一个 regime → 权重切换层（当前不存在）

## 五、遗留待决事项

### 待你拍板（涉及业务语义，不擅自改）

1. **是否引入 picks 自动下架机制（针对 O1）**——三种思路：
   - A. 写入前把昨天及之前的 `跟踪状态=🟢` 改为 `⚪ 已过期`，只今日入选保持 🟢
   - B. v6 改成 update 而非 append（同股+同日先 GET → 有则 PUT 更新，无则 POST 新建）
   - C. 加"持仓 vs 推荐"分离字段（飞书加新字段）

2. **是否启用因子权重校准管线（针对 O3）**——目前 IC 监控只输出诊断、不接通打分；要接通需要重构 `score_ai_relevance` / `score_valuation` / `score_trend` / `score_credibility` 4 个函数从 `factor_weights.json` 读权重而不是写死。

### 可继续清理（小工程量）
