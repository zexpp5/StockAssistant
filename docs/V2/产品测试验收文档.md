# StockAssistant 产品线路测试验收文档

更新时间：2026-05-15

本文档用于验证系统是否严格符合 [产品基线.md](产品基线.md)。任何涉及数据库、推荐逻辑、页面导航、数据拉取、运行状态的改动，都应按本文档做回归测试。

## 一、测试目标

核心目标：
- 验证新数据库必须从零开始，不迁移任何旧业务数据。
- 验证“我的池子”和“AI 助手”两条数据线没有混用。
- 验证 `watchlist` 只代表用户手动自选股。
- 验证系统拉取的科技/AI 股票池不会自动写入 `watchlist`。
- 验证系统科技池必须来自实际市场发现，不允许用静态种子池补齐固定数量。
- 验证 AI 推荐、AI 组合方案、已拉取股票池来自系统科技/AI universe，而不是来自手动自选股。
- 验证策略验证能基于历史推荐快照判断推荐是否准确，而不是只展示今天推荐什么。
- 验证深度研究只做买前解释和审查，不变成推荐列表、持仓编辑器或运行状态页。
- 验证运行状态能按美股/A股/港股说明每一步是否成功、数据落到哪里。
- 验证运行状态能展示数据时钟：计划时间、最近实际运行时间、下一次计划时间、每步耗时和数据落点。
- 验证数据库治理规则能防止重复数据、混线数据和过期数据被误当成可用数据。

当前数据库文件：

```text
stock_history_v2.duckdb
```

重要边界：
- 当前生产默认库为 `stock_history_v2.duckdb`。
- 旧 `stock_history.duckdb` 已删除，不再作为现状审查样本或回退来源。
- 后续新库重建时，旧 `watchlist/prices/picks/reviews/snapshots/discovery_history/earnings_history` 全部不迁移。
- 旧 `data/latest/*.json`、旧 cache、旧 report、旧 dashboard 产物不得作为新库初始化数据源。
- 如需保留旧文件，只能进入 archive/backup，生产链路不得读取。

推荐使用项目 Python：

```bash
/opt/homebrew/bin/python3
```

## 二、当前数据库对齐审查

本节只用于说明当前系统为什么需要重建，不能作为数据迁移清单。

当前数据库状态不在静态文档里写死。每天的实际数量应由检查脚本或验收脚本生成，避免文档隔天过期。
`system_universe` / `pool_membership` 的数量必须与当天实际市场发现一致；如果某市场抓空，不允许静态补数。

如需审查历史旧库，只能读取人工归档备份，不能读取生产根目录。当前生产检查命令：

```bash
/opt/homebrew/bin/python3 scripts/tools/current_state_report.py
```

后续可固化为：

```text
scripts/tools/current_state_report.py
scripts/tools/production_acceptance_check.py
```

当前基本判断：
- `prices` 当前包含系统科技/AI universe 的行情快照，是“已拉取股票池”的主要事实来源。
- `picks` 当前混合承载“自选股 AI 评级”和“生产推荐评分”，新库必须依靠 `universe_scope/source_origin` 区分来源。
- `reviews` 是 picks 后续表现跟踪，不定义股票属于哪个池。
- 当前新库必须显式有 `system_universe` 和 `pool_membership` 表，系统股票池身份不能再靠 `prices` 交集或页面文案去推断。

当前已发现的产品对齐风险：

1. **个股研究仍有旧 watchlist 入口**
   - 看板生成脚本里 `fetch_records_view()` 是 `watchlist LEFT JOIN latest prices`。
   - 所以“个股研究/主题展示”会只看到 `watchlist`，当前就只有英伟达。
   - 正确方向：个股研究应改成按 `code` 查询详情，可从 AI 推荐、已拉取股票池、自选股配置进入，而不是默认列出 `watchlist`。

2. **美股 production picks 当前仍偏 watchlist**
   - `daily_picks_v5.py` 当前打印和逻辑都显示“仅来自手动 watchlist”。
   - 这适合“自选股·AI 优选”，但不适合作为 AI 助手的系统池生产评分。
   - 正确方向：拆成两个模式：
     - `self_watchlist`：只服务自选股·AI 优选。
     - `tech_universe`：服务 AI 推荐/AI 组合方案。

3. **港股 picks 当前仍偏 watchlist**
   - `hk_picks.py` 当前候选说明是“仅来自手动 watchlist 港股”。
   - 这会导致没有手动港股时，港股生产 picks 为空。
   - 正确方向：港股 picks 默认应从 HK tech universe 生成，watchlist 只作为 overlay。

4. **A 股生产池方向相对正确**
   - `a_share_picks.py` 已支持 static/dynamic universe，并把 watchlist 作为 overlay。
   - 这是更符合产品基线的模式，美股和港股应向这个模式靠齐。

5. **缺显式股票池身份表**
   - 当前系统用 `prices ∩ universe` 推断“已拉取股票池”。
   - 这要求每个接口都重新实现一遍 universe 逻辑，容易出现页面和接口口径不一致。
   - P0 必须新增：

```text
pool_membership
  code
  name
  market
  pool_type        system_tech_universe | manual_watchlist | portfolio_observation
  source           us_universe | hk_universe | a_share_static | user
  active
  first_seen_at
  last_seen_at
```

v2 从零重建和验收命令：

```bash
/opt/homebrew/bin/python3 scripts/tools/init_stock_db_v2.py \
  --db stock_history_v2.duckdb \
  --seed-universe

STOCK_DB_PATH=stock_history_v2.duckdb \
/opt/homebrew/bin/python3 scripts/tools/current_state_report.py

STOCK_DB_PATH=stock_history_v2.duckdb \
/opt/homebrew/bin/python3 scripts/pipeline/fetch_stock_prices.py \
  --source tech-universe \
  --db-schema v2

STOCK_DB_PATH=stock_history_v2.duckdb \
/opt/homebrew/bin/python3 scripts/tools/build_v2_recommendations.py

STOCK_DB_PATH=stock_history_v2.duckdb \
/opt/homebrew/bin/python3 scripts/tools/production_acceptance_check.py \
  --allow-a-share-disabled

/opt/homebrew/bin/python3 scripts/tools/check_cutover_readiness.py
```

说明：
- `init_stock_db_v2.py` 只建新 schema 和代码定义的系统 universe，不读取旧 DuckDB、不读取旧 `data/latest`。
- `fetch_stock_prices.py --db-schema v2` 把新拉取行情写入 `price_daily`，不写旧 `prices`。
- `build_v2_recommendations.py` 只读取 `system_universe/pool_membership/price_daily`，写入 `recommendation_runs/recommendation_picks/portfolio_plans`。
- `production_acceptance_check.py` 在 v2 schema 下不再要求旧 `data/latest/plan_a_v5.json` 等 legacy 产物作为通过条件。

## 三、产品模块测试矩阵

### 1. 今日决策台

应回答：
- 今天系统能不能用。
- AI 推荐是否可参考。
- AI 组合方案是否存在。
- 已拉取股票池数量是否正常。

必须满足：
- 不生成新候选池。
- 不写 `watchlist`。
- 不写 `holdings`。
- 质量闸门或生产验收失败时，必须展示 FAIL/WARN，不允许伪装成可交易。

验收 SQL：

```sql
SELECT COUNT(*) FROM watchlist;
SELECT COUNT(*) FROM holdings;
```

操作今日决策台前后，上述计数不应因为打开页面而变化。

页面验收：
- 默认打开 `stock_dashboard.html` 应进入“今日决策台”。
- “AI 推荐”卡片应显示系统池候选数。
- “已拉取股票池”卡片应显示美股/A股/港股数量。
- 若 `production_acceptance_check.json` 为 FAIL，页面必须提示只读观察。

### 2. 我的池子 → 自选股配置

应回答：
- 我手动关注了哪些股票。

数据来源：
- 只允许 `watchlist`。

必须满足：
- 系统 universe 拉取不能自动增加 `watchlist` 行数。
- 用户手动新增/删除才改变 `watchlist`。
- 如果为了恢复页面而启用本地回放，必须在运行状态里显式标明 `cache_replay`，不能把回放数据当成实时行情成功。

验收 SQL：

```sql
SELECT code, name, market, source, created_at, updated_at
FROM watchlist
ORDER BY updated_at DESC;
```

负向测试：
- 执行系统池价格拉取后，`watchlist` 数量不应增加：

```bash
/opt/homebrew/bin/python3 scripts/pipeline/fetch_stock_prices.py --source tech-universe --dry-run
```

### 3. 我的池子 → 自选股·AI 优选

应回答：
- 我手动加入的股票里，哪些更值得优先看。

数据来源：
- 候选范围必须被 `watchlist` 限定。
- 可读取 `prices/picks/reviews` 做评分和回顾。

验收 SQL：

```sql
SELECT p.code, p.pick_date, p.model_source, p.signal
FROM picks p
LEFT JOIN watchlist w ON w.code = p.code
WHERE w.code IS NULL
  AND (
    p.model_source = 'self_watchlist'
    OR p.universe_scope = 'manual_watchlist'
  );
```

验收口径：
- 如果页面是“自选股·AI 优选”，不应展示不在 `watchlist` 的股票。
- 如果 `watchlist` 为空，该模块为空是正确状态。
- 不检查系统池生产 picks。`v6_cn/v6_us/v6_hk` 如果属于 `system_tech_universe`，不在 `watchlist` 是正确状态。

注意：
- 当前 `v6_us` 等 `model_source` 命名容易误导。新库必须用 `universe_scope/source_origin` 明确区分自选股模式和系统池模式。
- v2 初始化和行情抓取路径不得使用 `A_SHARE_STATIC_SEED` 之类的静态名单去补齐系统池。

### 4. AI 助手 → 已拉取股票池

应回答：
- 系统实际拉回了哪些科技/AI 股票。

数据来源：
- 系统科技/AI universe 与 `prices` 的交集。
- 不包含只存在于手动 `watchlist` 的股票。

API 验收：

```bash
curl -s http://127.0.0.1:8765/api/db/all-stocks
```

期望：
- 返回 `counts.total` 等于系统池实际拉取数量。
- 当前数量不应写死；应以 `system_universe` / `pool_membership` 的实时统计为准。
- 数量会随市场发现结果、数据源可用性、以及当日筛选规则变化。
- 验收的重点不是“固定几只”，而是“是否按美股 / A股 / 港股三市场的科技方向动态发现并落库”。
- 如果某市场今天发现为空，验收应显示该市场为空或降级，不应补成固定数量。
- 本地回放只允许作为恢复态，不允许替代 live market pull 的验收通过条件。

负向测试：
- 手动往 `watchlist` 加一只非科技股后，若它不在系统 universe 中，`/api/db/all-stocks` 不应出现它。
- 如果同一 ticker 同时存在于 `watchlist` 和系统池，已拉取股票池仍应显示 `_source_origin=system_pool`。

### 5. AI 助手 → AI 推荐

应回答：
- 系统科技/AI 股票池中，今天哪些更值得研究。

数据来源：
- `prices` 最新行情。
- 系统科技/AI universe 元数据。
- 可参考 `picks` 最新评分，但不能依赖 `watchlist` 才成立。
- 输出 `recommendation_runs`、`recommendation_picks`、`portfolio_plans`。

验收命令：

```bash
STOCK_DB_PATH=stock_history_v2.duckdb \
/opt/homebrew/bin/python3 scripts/tools/build_v2_recommendations.py
```

验收 SQL：

```sql
SELECT COUNT(*) FROM recommendation_runs;
SELECT COUNT(*) FROM recommendation_picks;
SELECT COUNT(*) FROM portfolio_plans;
```

验收口径：
- 即使 `watchlist` 只有 1 只，AI 推荐也应能基于系统池生成多只候选。
- `recommendation_runs.universe_scope` 和 `recommendation_picks.universe_scope` 应表达系统科技/AI 股票池，不应是 watchlist。

### 6. AI 助手 → AI 组合方案

应回答：
- 如果按系统推荐来构建组合，目标仓位和调仓建议是什么。

数据来源：
- AI 推荐。
- 生产 picks。
- 行情、风险指标、组合优化。
- 我的真实持仓只用于计算差额。

必须满足：
- 不自动写 `holdings`。
- 不自动写 `watchlist`。
- 页面必须明确这是模型目标组合，不是真实持仓。

验收文件：

```text
data/latest/plan_a_v5.json
data/latest/plan_v6.json
data/latest/trade_delta.json
data/latest/risk_metrics.json
```

验收 SQL：

```sql
SELECT category, name, MAX(taken_at)
FROM snapshots
WHERE category='pipeline'
GROUP BY category, name
ORDER BY name;
```

验收口径：
- 文件可以同步进 `snapshots`，但运行状态必须说明文件和 DuckDB 镜像各自是否存在。
- 如果文件缺失但 DuckDB 有旧镜像，页面只能作为 fallback 展示，运行状态仍应提示最新产物缺失。

### 7. AI 助手 → 策略验证

应回答：
- 过去 AI 推荐到底准不准。
- AI 组合方案有没有跑赢基准。
- 当前策略版本是否应该继续使用、观察、调整或暂停。

数据来源：
- 推荐运行记录 `recommendation_runs`。
- 推荐 point-in-time 快照 `recommendation_picks`。
- 组合方案 `portfolio_plans`。
- 推荐后续表现 `pick_outcomes`。
- 组合净值表现 `portfolio_performance`。
- 因子归因 `factor_attribution`。
- 策略复盘报告 `strategy_review_reports`。

当前过渡期可用数据：
- `recommendation_runs`
- `recommendation_picks`
- `portfolio_plans`
- `pick_outcomes`
- `portfolio_performance`
- `factor_attribution`
- `data/latest/recommendation_evidence.json`

页面验收：
- `AI 助手` 下应有 `策略验证` 入口。
- 页面必须展示策略版本、样本数、Top N 胜率、alpha、最大回撤、结论。
- 样本不足时只能显示“继续观察”，不能给强结论。
- 策略验证页面不得直接输出当天买入/卖出清单。

样本阈值验收：
- `sample_size < 20`：只能显示“继续观察”。
- `20 <= sample_size < 60`：只能给弱结论，并提示样本不足。
- `sample_size >= 60`：可以给策略健康判断。
- 样本阈值必须按市场分别计算；美股、A 股、港股不能混在一起给强结论。

数据验收 SQL：

```sql
SELECT COUNT(*) FROM recommendation_runs;
SELECT COUNT(*) FROM recommendation_picks;
SELECT COUNT(*) FROM strategy_review_reports;
```

新库完成后的验收 SQL：

```sql
SELECT strategy_version, period_type, period_start, period_end, sample_size, conclusion
FROM strategy_review_reports
ORDER BY period_end DESC;

SELECT run_id, strategy_version, model_version, universe_scope, data_cutoff_at, generated_at
FROM recommendation_runs
ORDER BY generated_at DESC
LIMIT 20;
```

硬性验收：
- 每条推荐必须能追溯到 `run_id`。
- 每个 `run_id` 必须能看到 `strategy_version`、`model_version`、`universe_scope`、`data_cutoff_at`。
- 后续表现只能基于历史快照计算，不允许用后来补齐的数据重写当时推荐理由。
- 策略版本调整必须新增版本，不得覆盖旧版本。

负向测试：
- 删除或缺失样本数时，页面不得显示“策略有效”。
- 当前策略表现 FAIL 时，今日决策台只能提示风险，不得隐藏策略失效信号。
- 系统不得自动切换生产策略版本。

### 8. 深度研究 → 个股研究

应回答：
- 单只股票为什么值得研究、贵不贵、增长是否兑现、风险是什么。

正确数据来源：
- 可以从 `watchlist`、AI 推荐、已拉取股票池进入。
- 单只详情应通过 `code` 查询 `watchlist/prices/picks/reviews/recommendation_runs/recommendation_picks/earnings_history`。

API 验收：

```bash
curl -s http://127.0.0.1:8765/api/db/stock-history/NVDA
```

当前 NVDA 应返回与新库事实一致的多源明细；其中 `recommendation_runs` / `recommendation_picks` 应能追溯到 AI 推荐现场，`watchlist` 只代表手动自选，不代表系统池身份。

负向测试：
- 个股研究页面不应默认只列 `watchlist`。
- 页面必须能区分“用户自选”“系统推荐”“真实持仓”三种身份。

### 9. 深度研究 → 产业链地图

应回答：
- 股票处在科技/AI 产业链的哪一段。

数据来源：
- 自选股研究字段：`watchlist.chain/chain_tier/chain_role/layman_intro`。
- 系统池元数据：后续应来自 `pool_membership` 或 universe metadata。

当前风险：
- `classify_watchlist_chains.py` 当前只更新 `watchlist`。
- 运行日志显示“尝试 UPDATE 89 条，实际匹配 1 条”，说明它把系统池候选当成 watchlist 去更新，只有 NVDA 匹配。

验收口径：
- 产业链地图如果展示系统池股票，不能依赖 `watchlist` 才有产业链字段。
- 系统池产业链元数据应独立存储或由 universe 提供。

### 10. 管理 → 运行状态

应回答：
- 美股/A股/港股每条链路是否成功。
- 每一步数据落到了哪里。
- 哪一步失败、影响是什么、下一步怎么处理。
- 计划任务什么时候该跑，最近一次实际什么时候跑。

验收文件：

```text
data/latest/recommendation_quality_gate.json
data/latest/production_acceptance_check.json
data/latest/source_health.json
data/latest/recommendation_evidence.json
data/latest/pipeline_status_production.json
data/latest/pipeline_status_research.json
data/latest/pipeline_status.json  # 兼容别名：只代表 production，不由 research 覆盖
daily_refresh.log
```

验收口径：
- 必须分市场展示，不能只有一个总 FAIL。
- 每个步骤必须有“数据落点”。
- `prices/picks/reviews/data/latest/*.json` 必须有解释。
- 运行状态不输出买卖建议。
- `recommendation_evidence.json` 只能作为可信度提示，不应单独阻断今日入口；今日入口是否只读，以质量闸门和生产验收为准。

数据时钟验收：
- 页面必须显示 `08:30 早盘主线`、`16:30 A/H 收盘线`、`21:00 增强研究线`、`周频策略验证`。
- 每条任务线必须显示计划时间、最近实际运行时间、最近状态、覆盖范围。
- `data/latest/pipeline_status_production.json` 与 `data/latest/pipeline_status_research.json` 必须分线记录。
- `data/latest/pipeline_status.json` 是兼容别名，只能指向 production/morning/a_share_only 最新状态；research 失败不能覆盖它。
- 每个 pipeline status 必须包含：

```text
run_id
mode
status
status_role
started_at
updated_at
completed_at
schedule[]
steps[]
failed_steps[]
slowest_steps[]
```

验收命令：

```bash
bash -n daily_refresh.sh
/opt/homebrew/bin/python3 -m py_compile scripts/pipeline/build_stock_dashboard_html.py
/opt/homebrew/bin/python3 scripts/pipeline/build_stock_dashboard_html.py
rg -n "数据时钟|最近运行步骤|跑批状态" stock_dashboard.html
```

负向测试：
- 没有 `pipeline_status.json` 时，页面仍应展示计划时间表，而不是空白。
- 某步骤失败时，运行状态必须显示失败步骤、数据落点和下一步建议。
- 运行状态不得把失败步骤解释成买卖建议。

状态分类验收：
- DuckDB 被占用时显示 `duckdb_locked`。
- 外部源不可用时显示 `source_down`。
- 外部源限流或部分字段失败时显示 `source_degraded`。
- 主源失败但 fallback 成功时显示 `fallback_ok`。
- 市场休市时显示 `market_closed`。
- 当天本不应产生数据时显示 `data_not_expected`。
- `fallback_ok`、`market_closed`、`data_not_expected` 不应被粗暴算成阻断级 FAIL。

### 11. 数据库治理、去重与新鲜度

应回答：
- 同一行情、同一财报、同一推荐重复拉取时，数据库会不会重复。
- 数据是不是及时，还是旧数据被误当成今天可用。
- 每条数据到底属于哪个交易日、数据源什么时候发布、系统什么时候拉回。

必须有的时间字段口径：
- `business_date` / `trade_date`：业务日期或交易日。
- `source_updated_at` / `reported_at`：数据源发布时间。
- `fetched_at`：系统拉取时间。

去重验收口径：

| 数据 | 唯一键 | 期望 |
| --- | --- | --- |
| 股票池成员 | `pool_id + market + symbol` | 同池同股票不重复 |
| 日行情 | `market + symbol + trade_date + interval` | 重复拉取 upsert |
| AI 推荐 | `strategy_version + model_version + universe_scope + run_date + symbol` | 同运行事实不重复 |
| 组合方案 | `run_id + plan_version + strategy_scope + symbol` | 绑定推荐运行 |
| 财报 | `market + symbol + fiscal_year + fiscal_quarter + statement_type + source` | 内容相同跳过，内容变化生成版本 |
| 13F | `institution + cik + report_period + symbol` | 同报告期 upsert |
| 事件日历 | `market + symbol + event_type + event_date + source` | 重复事件不新增 |
| 运行步骤 | `run_id + step_name` | 每次运行 append |

新鲜度验收：
- 美股日行情：美股收盘后超过 36 小时没更新，运行状态标记过期。
- A 股/港股日行情：16:30 收盘线后仍未更新当天交易日，标记过期。
- AI 推荐：当天 08:30 主线未生成，标记缺失。
- A 股/港股 picks：16:30 收盘线后未生成，标记该市场推荐缺失。
- 组合方案：推荐已更新但组合未更新，标记组合过期。
- 财报：财报季后仍缺最新季度，标记财报覆盖不足。
- 策略验证：成熟样本未更新，标记策略验证滞后。

市场日历验收：
- 美股推荐必须显示使用的是哪个美股交易日收盘数据。
- A 股 16:30 任务必须先判断当天是否 A 股交易日，以及收盘数据是否可用。
- 港股必须按香港交易所交易日和假期判断是否应拉取。
- 节假日、半日市、未开市不应被误判为数据缺失。
- 市场关闭时，运行状态应显示 `market_closed` 或 `data_not_expected`。

负向测试：
- 重跑同一天行情拉取，`price_daily` 不应出现重复行。
- 重跑同一季度财报拉取，内容未变化时不应新增当前记录。
- 同一推荐 `run_id + symbol` 不应重复插入。
- 旧组合方案不能因为有 DuckDB fallback 就被标记为最新可用。

## 四、数据库层级与重建切换验收

### 新库重建硬验收

执行新库重建前必须确认：

- 新库文件为空库或全新文件。
- 没有从旧 DuckDB 执行 `INSERT INTO new_db SELECT ... FROM old_db`。
- 没有从旧 JSON/cache/report 初始化业务表。
- `manual_watchlist` 为空，除非用户重新手动添加。
- `holdings` 为空，除非用户重新确认真实持仓。
- `system_universe` 从代码定义或新配置生成，不从旧 `prices` 反推。
- `price_daily` 从新拉取行情生成。
- `recommendation_runs/recommendation_picks` 从新策略运行生成。
- `strategy_review_reports` 从新快照和后续表现自然累积，不回填旧推荐。

阻断项：
- 任何旧表旧产物被导入新库，都视为测试 FAIL。
- 新库页面显示旧日期推荐、旧组合、旧策略复盘，视为测试 FAIL。
- 系统从 archive/backup 读取生产数据，视为测试 FAIL。

### 新库切换流程验收

必须按以下流程验收：

1. 归档当前 DuckDB 和 `data/latest`，只读保存。
2. 创建新库，例如 `stock_history_v2.duckdb`。
3. 用环境变量或配置显式指向新库。
4. 初始化 schema，不从旧库复制业务数据。
5. 从系统 universe 重新生成 `system_universe` 和 `pool_membership`。
6. 重新拉取行情、生成推荐、组合、运行状态。
7. 验证 dashboard/API 全部读取新库。
8. 禁止 fallback 读取旧 `data/latest`、旧 cache、旧 snapshots。
9. 旧库只读归档，不允许生产代码读取。

阻断项：
- 新库失败后 API 或 dashboard 自动 fallback 到旧库，测试 FAIL。
- 新库缺产物时页面读取旧 `data/latest`，测试 FAIL。
- 未显式配置新库路径就开始写入，测试 FAIL。

当前物理表大体可用，但以下身份层是 P0，不是可选建议。

目的：

## 五、数据拉取性能审查

当前已经有的优化：
- `fetch_stock_prices.py` 已使用 yfinance 批量历史价下载。
- `fetch_stock_prices.py` 已给 yfinance `info` 做本地 TTL cache。
- A 股 picks 和港股 picks 已有并发 worker 和当天因子缓存。
- `daily_refresh.sh` 已开始按任务线生成 `data/latest/pipeline_status_<mode>.json`，并把 production/morning/a_share_only 同步到 `pipeline_status_production.json` 和兼容别名 `pipeline_status.json`；research 只写 `pipeline_status_research.json`。
- 运行状态页已开始展示数据时钟和最近运行步骤。

当前主要慢点：

1. **事件日历**
   - 日志显示 `event_calendar_daily` 处理约 289 条增减持/事件数据，进度条持续数分钟。
   - 这一步对每日早盘“能不能看 AI 推荐”不是强阻断，应改成低优先级或增量缓存。

2. **政策事件扫描被重复调用**
   - `policy_scan_daily.py` 会调用 `scan_recent_policies(days=7)`。
   - `themes_under_policy_tailwind(days=14)` 内部又再次调用 `scan_recent_policies(days=14)`。
   - 这会重复抓新闻源，建议复用第一次扫描结果或落缓存。

3. **美股/港股 production picks 未按系统池跑**
   - 当前美股 picks 只跑 watchlist，很快但产品口径不对。
   - 后续改成系统池后必须先设计缓存和批量拉取，否则会明显变慢。

4. **daily_refresh 串行步骤过多**
   - 很多步骤并不互相依赖，例如 13F、OpenBB、IPO、事件日历、政策扫描、HTML 重建前的部分产物可以拆成主链路和低优先级链路。

建议优化优先级：

```text
P0：把 daily_refresh 拆成主链路和增强链路
    主链路：价格 → AI 推荐 → 组合方案 → 质量闸门 → HTML/今日决策台
    增强链路：13F、IPO、事件日历、政策、OpenBB、深度研究补充

P0：拆分 daily_refresh 为 08:30 / 16:30 / 21:00 三条任务线，并沿用 pipeline_status 记录状态

P1：政策扫描复用同一批 news_items，避免 days=7 和 days=14 重复抓取

P1：事件日历按日期增量缓存，非交易日或当天已生成时直接复用

P1：美股/港股 production picks 改成 system universe，但必须先做因子缓存和限流

P2：运行状态页展示每步耗时、最近成功时间、一键重跑命令

P2：新增策略验证页面，直接以 recommendation_runs/recommendation_picks/pick_outcomes 为主，历史旧表只作归档参考
```

## 七、上线前阻断项

以下任一情况出现，都不应把系统标记为“可交易”：

- AI 推荐依赖 `watchlist` 才能生成。
- 已拉取股票池混入只存在于 `watchlist` 的手动股票。
- 自选股·AI 优选展示了不在 `watchlist` 的股票。
- AI 组合方案自动写入 `holdings`。
- 今日决策台或 AI 推荐在质量闸门 FAIL 时仍显示“可买/可交易”。
- 运行状态无法说明失败步骤的数据落点。
- 个股研究只能显示 `watchlist`，无法从 AI 推荐/已拉取股票池进入单只详情。
- 数据时钟缺失，用户无法知道计划任务和最近实际运行时间。
- 策略验证使用最新数据回填历史推荐，导致未来函数。
- 策略验证没有样本数却输出强结论。
- 系统自动切换策略版本或自动修改真实持仓。
- 重复拉取财报、行情、推荐后产生重复当前记录。
