# AI 与科技成长推荐规则

日期：2026-06-09

状态：产品规则草案，可作为后续推荐系统、股票池分层、页面展示和策略验证的对齐依据。

评审修订：2026-06-09

本次评审后的核心结论：

```text
方向正确，但不能直接把三周期权重写成生产公式。
P0 只做分层、准入、红旗和页面表达，不碰现有评分公式。
三周期先做分项展示和回测标签，等历史回算 + 样本外验证通过后，再决定是否合成新分数。
```

必须遵守的开发红线：

1. **不秘密改公式**：任何新权重、新总分、新 rating 阈值，都必须先经过数据卫生、消融、历史回算、样本外验证和用户确认。
2. **不双引擎**：已有 `data_usability gate`、`quality_tag`、`premarket_gate`、`strategy_eval.py` 等单一来源逻辑必须复用，不能平行再造。
3. **先 filter/tag，后 score**：短期内只新增身份标签、推荐资格、红旗拦截和展示字段，不新增未验证的加权总分。
4. **证据字段必须有来源**：订单、AI 收入、backlog、CapEx 受益等字段必须写清数据来源；拿不到数据时降级为 `research_only` 或 `watch_only`，不能让模型猜。
5. **美股、A 股、港股都要有口径**：本规则当前示例以美股为主，但落地时必须覆盖系统已有多市场推荐线，不允许只修美股、留下 A/HK 口径悬空。

## 一、这份文档解决什么问题

本系统不能继续只做“分数高就推荐”。科技成长股，尤其 AI 相关股票，必须先判断身份和阶段，再看订单、估值、买点和风险。

本规则的目标是把 AI 推荐从“热点榜单”升级为：

```text
科技成长股票筛选 + 阶段识别 + 周期视角 + 风险拦截 + 可验证推荐
```

它解决以下问题：

1. 避免非 AI/非科技核心股票因为估值便宜、反转分高而误入 AI 推荐。
2. 避免 AVGO、MRVL 这类 AI 核心芯片股，以及 HPE、DELL 这类 AI 基建股，在过热阶段被无脑追高。
3. 让 VST、GEV、VRT 这类数据中心电力/散热受益股能被正确识别，而不是被排除在 AI 主线之外。
4. 让 ADSK、INTU、CRM、ADBE 这类软件股进入“AI 集成/应用阶段”观察，而不是和芯片股混在同一个短线榜单里。
5. 让系统明确区分“今晚能不能买”“未来半年是否在主线”“长期是否值得跟踪”。

## 二、系统定位

StockAssistant 的推荐系统长期定位不是“AI 概念股推荐器”，而是“科技成长轮动系统”。

当前 AI 是最强主线，但未来科技成长主线可能在以下方向之间轮动：

```text
半导体
云计算
AI 算力
AI 网络
AI 应用软件
数据中心电力
液冷和基础设施
网络安全
工业自动化和机器人
医疗科技
互联网平台
金融科技
消费科技
```

因此系统必须保留科技成长大池，但在不同阶段识别不同受益链条。

## 三、当前产业阶段判断

截至 2026-06-09，本系统对 AI 产业阶段的判断为：

```text
AI 数据中心建设后半段
+ AI 推理/Agent/企业集成商业化前半段
```

这意味着：

1. 未来 1-8 周更看重市场风险、回调支撑、盘前环境和事件催化。
2. 未来 3-12 个月更看重订单、收入兑现、CapEx 受益和估值是否透支。
3. 未来 1-5 年更看重平台能力、数据、客户、生态、护城河和现金流。

系统推荐必须同时输出短期、中期、长期三个视角，而不是只给一个“强买”。

注意：短期、中期、长期三个视角在 P0/P1 阶段只做**分项展示和验证标签**，不得直接合成新的生产总分。

### 1. 阶段判断必须结构化

产业阶段不能只写在文档散文里，也不能被代码当成永久事实。它必须作为一个带日期、版本、证据和复核节奏的结构化输入。

示例：

```text
market_phase_snapshot:
  as_of: 2026-06-09
  phase_id: ai_infra_buildout_to_inference
  phase_name: AI 数据中心建设后半段 + 推理/Agent/企业集成前半段
  scope: US / global_tech
  confidence: medium
  evidence:
    - 大厂 CapEx 仍高
    - NVDA/AVGO/MRVL 等 AI 核心芯片/网络收入仍在兑现
    - 数据中心电力和网络需求继续上行
  review_cycle: weekly
  next_review_at: 2026-06-16
  owner: human_review_required
```

> 上例 `evidence` 里的公司名（NVDA/AVGO/MRVL 等）同 §4 示例声明：仅为截至该 as_of 的说明样本，不是写死的生产归属；分类以版本化 `tech_growth_layers` + evidence snapshot 为准。

硬规则：

```text
代码引用 phase_id，不引用文档散文。
阶段变化只能影响展示、解释和研究优先级，不能偷偷改生产评分公式。
阶段快照必须保存 PIT，用于回看当时为什么这么判断。
如果阶段快照过期，相关页面必须提示“阶段判断待复核”。
```

> 实现注意：`market_phase_snapshot` 是本规则**强制新增的阶段快照实体**。股票分层可以落在 `system_universe` 新列，也可以落在独立 `growth_layer/tech_growth_layers` 表；无论采用哪种实现，都不得复用或污染 `pool_membership.pool_type`。§12.1 要保存的 `market_phase_snapshot_id` 即引用阶段快照。

### 2. 动态调整原则

一只股票今天不好，不代表永远不好；今天很强，也不代表一直能买。系统必须允许股票在不同动作之间动态流转。

注意：`eligibility`（能不能进候选）与 `action`（该怎么做）是两个独立的轴——一只票可以同时 `eligibility=buyable` 且 `action=wait_entry`。下面的流转是两轴的**组合态**示意，实现时不要塌缩成单一状态机。

典型流转：

```text
blocked      -> wait_entry      风险红旗解除，但买点仍需确认
wait_entry   -> focus_research  回调支撑出现 + 证据仍成立
focus_research -> research_only 财报/订单证据不足或数据源失效
research_only -> buyable候选    证据补齐 + 数据可用 + 无硬红旗
watch_only   -> research_only   主题相关性增强，出现公司级证据
buyable候选  -> blocked         盘前风险、过热、估值或急跌红旗触发
```

动态调整必须留下原因：

```text
上一状态
新状态
触发条件
数据来源
as_of 时间
是否影响短期/半年/长期视角
```

晋升条件示例：

```text
watch_only -> research_only:
  出现公司级证据，例如财报、订单、客户、产品或产业链证据。

research_only -> wait_entry:
  数据可用性通过 + 证据仍成立 + 无 all-scope 红旗。

wait_entry -> focus_research:
  价格回调有支撑 + 盘前/宏观风险未拦截 + 买前研究值得展开。
```

降级条件示例：

```text
focus_research -> wait_entry:
  逻辑仍成立，但价格过热、估值偏满或买点消失。

wait_entry -> blocked:
  短期急跌、盘前橙红、财报事件风险或过热红旗触发。

research_only -> watch_only:
  证据过期、订单未兑现、主题相关性减弱。

任何状态 -> excluded:
  身份确认不属于科技成长主线，或数据源被证明错误。
```

### 3. 证据有效期与个股复核节奏

证据不是确认一次就永久有效，必须带有效期，过期自动降级（对应 `evidence_status` 的 stale）：

```text
财报/分部收入证据      ≤ 1 个季度（下次财报发布即需复核）
订单/backlog/指引证据  ≤ 1 个季度或至下次指引更新
新闻/事件证据          ≤ 30 天（仅作线索，不单独支撑 confirmed）
CapEx/供应链受益证据   ≤ 2 个季度
超期未更新 → evidence_status 自动转 stale → eligibility 降为 research_only
```

复核节奏（半自动，呼应 §3.1 的 review_cycle）：

```text
每日：价格、涨跌幅、过热、盘前风险、买点 → 刷新 short_term 状态与 action
每周：产业阶段 market_phase_snapshot 复核
每次财报后：强制复核该股 evidence_status 与基本面证据
分类基线：手审白名单变更走 PR + classification_version 递增，不随价格自动改
```

### 4. 主动防漏机制

系统不能等用户发现问题才补规则。每一次推荐都必须进入“生成 → 拦截 → 复盘 → 归因 → 改进候选”的闭环。

每日自动检查：

```text
盘前：premarket_gate 是否橙/红，若是则今日新开仓动作降级
盘中/收盘后：今日推荐里是否出现急跌、过热、放量下跌、财报异常
推荐列表：是否有非科技成长、ETF-only、证据过期股票混入
持仓叠加：真实持仓是否触发集中度或纪律线提醒
```

每周自动复盘：

```text
漏推荐：本周明显上涨的科技成长股，系统为什么没提前进入候选
误推荐：进入 focus_research / wait_entry 后下跌的股票，错误来自身份、证据、估值、买点还是盘前环境
过热样本：涨幅过高后回撤的股票，是否应被更早拦截
行业轮动：AI 核心、AI 基建、软件、电力、平台哪个分层在变强或变弱
```

每月策略复盘：

```text
按分层、动作、周期看中位数 alpha、胜率、最大亏损
找出最稳定的正贡献规则，以及最稳定的负贡献规则
只允许把“连续验证有效”的规则升级为生产候选
任何新公式、新权重、新阈值都必须进入 strategy_eval 样本外验证
```

问题入库要求：

```text
每个漏推荐/误推荐案例都必须记录：
  symbol
  as_of
  当时分层
  当时 evidence_status
  当时 eligibility/action
  后续 1d/5d/20d 表现
  错误归因：数据缺失 / 身份误判 / 证据过期 / 买点错误 / 盘前风险 / 估值过热 / 其它
  是否形成规则改进候选
```

> 落点：问题入库需要一张独立表（如 `recommendation_review_log`，与 `market_phase_snapshot` 同属本规则新增实体），不得写入 picks/holdings；归属 P3（周/月复盘上线时建），不阻塞 P0。

原则：案例可以推动规则候选，但不能直接推动生产公式变更。生产公式只能由验证通过的候选推动。

## 四、股票池分层

系统不再使用一个混合榜单直接推荐。所有股票必须先进入分层。

> **示例只是说明，不是生产规则。** 本章及全文出现的公司名（NVDA / AVGO / MRVL / VST 等）仅为**截至 2026-06-09 的分类示例与验收样本**，公司归属未被永久写死。
> - 生产系统**不得从本文档硬编码公司归属**，必须读结构化字段 `primary_layer / secondary_layers_json / evidence_status`。
> - 分类机制是**半自动**：手审白名单/公司主业 → 定基础分类；证据/行情/财报/订单 → 动态拉取复核；推荐资格与动作 → 每日计算；每次结果 → 保存 PIT 版本（`tech_growth_layers` 版本号 + evidence snapshot）。
> - 动态拉取的是证据、行情、估值、财报、订单、主题阶段；分类**不靠价格自动算**（会重演 VIPS 误分类），也**不靠文档写死**（会过期）。
> - 举例：AVGO `primary_layer=ai_core, secondary_layers=["ai_network","custom_silicon"]`；MRVL 同理（定制芯片 + 光互联/高速连接）——属 ai_core，不属 ai_infrastructure。

分层必须支持多标签。很多公司天然跨层，例如：

```text
GOOGL:
  primary_layer = internet_platform
  secondary_layers = ["ai_core", "cloud_ai"]

VST:
  primary_layer = power_datacenter
  secondary_layers = ["ai_infrastructure"]
```

因此代码上不能只用一个 `tech_growth_layer` 单值枚举解决全部问题。推荐字段应至少包含：

```text
primary_layer
secondary_layers_json
layer_confidence
```

如果暂时只能存单值，必须写明优先级规则，不能让同一股票在不同页面被不同逻辑误标。

### 1. AI 核心池

含义：
- 直接提供 AI 算力、AI 网络、AI 芯片、AI 云平台或 AI 基础模型能力。

示例：

```text
NVDA
AVGO
MRVL
TSM
AMD
MU
CRDO
ALAB
MSFT
GOOGL
META
AMZN
```

推荐意义：
- 当前 AI 主线的核心观察池。
- 可以进入 AI 推荐和 AI 组合方案，但必须通过估值、订单、红旗和买点检查。

### 2. AI 基建池

含义：
- 受益于 AI 数据中心建设的服务器、硬件与系统集成商，但不直接提供 AI 芯片或模型。

示例：

```text
HPE
DELL
SMCI
```

推荐意义：
- 当前阶段重要，但需要和 AI 核心池分开评分。
- 重点看订单、backlog、数据中心收入、毛利率、交付能力和估值透支。
- 电力/电网/散热等基础设施另立「电力/数据中心基础设施池」（见本节第 6 条），不混入本池。

### 3. 科技软件池

含义：
- AI 进入企业工作流后可能受益的软件、设计、财税、办公、CRM、安全等公司。

示例：

```text
ADSK
INTU
CRM
ADBE
NOW
PANW
CRWD
SNOW
DDOG
```

推荐意义：
- 更偏 6-18 个月和长期逻辑。
- 不应因为短期 AI 基建热度较弱就被忽略。
- 重点看 AI 功能是否带来付费率、留存率、ARPU、利润率或新产品增长。

### 4. 互联网平台池

含义：
- 拥有流量、数据、广告、搜索、电商、本地生活或内容分发能力的平台公司。

示例：

```text
GOOGL
META
AMZN
NFLX
UBER
SHOP
```

推荐意义：
- 重点看 AI 是否改善广告、搜索、推荐、云、客服、内容生产和成本结构。
- 不能只因为“互联网 + AI”标签就进入 AI 核心推荐。

### 5. 主题观察池

含义：
- 通过 ETF、新闻主题或宽泛行业线索发现，但尚未证明属于可买科技成长主线。

示例来源：

```text
KWEB
PAVE
ICLN
REMX
URA
LIT
```

推荐意义：
- 只能观察，不能直接推荐买入。
- ETF 持仓只能作为发现入口，不能作为“AI 相关性”的最终证据。

规则：

```text
仅 ETF 来源 + 无第二证据 = watch_only
宽泛行业主题 + 无订单/业务证据 = research_only
非科技/非成长/非主线 = excluded
```

### 6. 电力/数据中心基础设施池

含义：
- 为 AI 数据中心提供电力、电网、配电、散热/液冷等基础设施，受益于 CapEx，但不属于芯片或服务器链。

示例：

```text
VST
GEV
VRT
ETN
PWR
```

推荐意义：
- 对应 `primary_layer = power_datacenter`，与 AI 基建池分开评分。
- VST/GEV/VRT 等可带 `secondary_layers = ["ai_infrastructure"]`，但身份归属以 power_datacenter 为准。
- 重点看数据中心电力/散热订单、backlog、利用率、交付能力和估值透支。

## 五、推荐准入门槛

任何股票进入推荐前，必须先过准入门槛。

### 1. 行业身份门槛

系统必须明确回答：

```text
这家公司到底属于哪个行业？
它是不是科技成长方向？
它是不是当前阶段的受益者？
```

如果无法回答，不能进入推荐，只能进入观察。

### 2. AI/科技相关性门槛

AI 相关性不能只靠名字、新闻热词或 ETF 标签。

有效证据包括：

```text
财报中披露 AI 相关收入
管理层在财报电话会明确说明 AI 订单或客户需求
客户名单或供应链订单可验证
backlog 或指引上修来自 AI/数据中心需求
CapEx 直接流向该公司所在链条
产品被用于 AI 训练、推理、网络、数据中心、电力、散热或企业工作流
```

无效或弱证据包括：

```text
只在 ETF 中出现
只在新闻标题里出现 AI
只属于宽泛互联网或消费行业
只因为股价上涨被市场贴 AI 标签
```

### 3. 订单/业绩确认门槛

系统必须优先确认：

```text
收入是否增长
AI/数据中心收入是否增长
订单是否增长
backlog 是否增长
指引是否上修
毛利率是否稳定
客户是否真实
```

只讲故事、没有收入或订单确认的股票，不允许给“重点研究以上”的动作。

### 4. 估值不过度透支门槛

系统必须判断：

```text
PE / Forward PE 是否极端
PS / EV/Sales 是否极端
PEG 是否合理
估值是否高于同业太多
增长率能否支撑估值
```

好公司估值过满，也只能等买点，不能直接推荐。

### 5. 技术面和回调支撑门槛

系统必须判断：

```text
是否刚经历过急涨
是否处于过热区
是否出现放量下跌
是否跌破关键支撑
回调后是否开始企稳
相对强度是否仍优于行业
```

技术面判断不得重新手写一套模糊规则。优先复用现有逻辑：

```text
quality_tag：stock_research/core/quality_tag.py
价格行为/跌深反转 gate：build_v2_recommendations.py 中现有 price-action 相关 gate
盘前风险：stock_research/core/premarket_gate.py
```

如果后续需要新增阈值，必须写清：

```text
使用哪条均线
阈值是多少
对应哪个市场
对应哪个持有周期
是否经过历史回算
```

原则：

```text
不是越跌越买。
不是越涨越追。
回调必须有支撑，反弹必须有确认。
```

## 六、三周期视角

系统每只股票不再只有一个解释不清的总分，而是至少输出三个周期视角。

重要修订：

```text
本章字段在 P0/P1 阶段只作为分项展示和回测标签。
不得直接按本章文字生成新的加权公式。
不得替代现有 build_v2_recommendations.py 的生产评分。
```

原因：

```text
系统已经因为手拍权重吃过亏。
momentum 降权、reversal 启用，必须来自 IC 审计和消融验证。
三周期权重如果未经验证直接上线，会重犯“分数高就推荐”的旧错误。
```

建议字段：

```text
short_term_view     今晚/本周是否适合买，只做展示和动作解释
six_month_view      未来半年是否处在主线，只做阶段判断
long_term_view      一年以上是否值得长期跟踪，只做质量判断
```

### 1. 短期视角

适用问题：

```text
今晚能不能买？
本周能不能加仓？
急跌后能不能接？
```

短期视角展示项：

```text
盘前闸门状态
技术面和回调支撑
事件催化
估值是否过热
订单/基本面是否已有确认
持仓和仓位风险
```

短期视角主要服务交易节奏，不代表长期价值。

盘前风险只使用现有 `premarket_gate` 的结论作为 gate，不进入新评分，避免重复计算。

### 2. 半年视角

适用问题：

```text
未来 3-12 个月是否还在主线？
当前产业阶段是否继续受益？
```

半年视角展示项：

```text
订单/客户/收入确认
AI 数据中心或 AI 集成受益程度
估值是否透支
回调是否有支撑
财务质量和竞争格局
```

半年视角是当前系统最重要的研究视角，但在验证完成前不直接合成生产评分。

### 3. 长期视角

适用问题：

```text
这家公司 1-5 年是否仍可能是科技成长核心资产？
```

长期视角展示项：

```text
平台能力/护城河
客户和生态
数据/产品壁垒
长期利润率和现金流
行业空间
资产负债和资本开支风险
```

长期视角不能直接等于“今晚可以买”。

### 4. 什么时候才允许变成分数

只有同时满足以下条件，才允许把三周期视角合成为 `score_short_term`、`score_6m`、`score_12m_plus`：

```text
字段数据源稳定
PIT 快照可回放
历史回算中位数 alpha 改善
胜率改善
最大亏损下降
不是靠少数极端赢家
样本外前瞻验证通过
用户确认可以进入生产策略
```

## 七、总分和动作定义

系统可以保留现有 100 分制和历史 `rating` 字段，但动作必须新增一层解释字段，不能直接用新语义覆盖旧字段。

现有系统仍有：

```text
rating >= 75  -> strong_buy
rating >= 60  -> buy
```

这套旧字段牵涉 DB、页面、策略验证和历史样本，不能突然改含义。

短期迁移方案：

```text
旧 rating：保留，用于兼容历史页面和验证。
新 action：新增，用于表达用户动作。
新 eligibility：新增，用于表达是否有资格进入推荐。
新 red_flags：新增，用于表达为什么被降级或拦截。
```

推荐动作必须使用以下枚举：

```text
focus_research      重点研究
wait_entry          等买点
watch_only          只观察
research_only       研究，不进真钱
blocked             红旗拦截
exclude             剔除
```

前端展示必须使用中文，不得裸露英文枚举。

```text
机器枚举          中文展示      一句话解释
focus_research   重点研究      值得进入买前研究，不等于买入。
wait_entry       等买点        逻辑成立，但价格、风险或确认条件还不合适。
research_only    仅研究        数据或证据不足，不进真钱候选。
watch_only       只观察        可以关注，但不参与推荐和组合。
blocked          暂时拦截      触发风险红旗，等条件改善再看。
exclude          剔除          身份不符合系统范围，或数据明显错误。
buyable          可进入候选    仅表示有资格进入候选，不代表自动买入。
```

注：`buyable` 属于 `eligibility`（资格）字段，不是 `action`；列在此处仅为集中解释中文展示词。

语义边界：

```text
exclude：身份不匹配、非科技成长、非目标市场、数据源明显错误。
blocked：本来可能值得看，但触发推荐本体红旗，例如过热、盘前风险、估值极端。
watch_only：可以观察，但不能进入真钱候选。
research_only：可以研究，但缺证据或缺数据，不能直接交易。
wait_entry：逻辑成立，但买点未到。
focus_research：进入买前研究，不等于买入指令。
```

禁止只输出：

```text
强买
买入
无脑加仓
```

说明：文档中的动作不是券商意义上的投资评级，也不是自动交易指令。

## 八、红旗按作用范围拦截

红旗优先级高于总分。触发红旗时，即使分数很高，也不能给买入建议；但必须说明红旗只影响哪个周期或哪个页面。

但红旗必须带作用范围，不能把短期风险误用成长期否决。

```text
scope = short_term       只影响今晚/本周买点
scope = six_month        影响 3-12 个月配置判断
scope = long_term        影响长期跟踪资格
scope = all              全期限否决
scope = portfolio_overlay 只在真实持仓/组合层叠加，不属于推荐本体
```

红旗作用范围建议：

```text
identity_mismatch       -> all
data_usability_failure  -> all / research_only
premarket_high          -> short_term
overheated              -> short_term / six_month
valuation_extreme       -> six_month，可保留长期观察
order_unconfirmed       -> six_month / research_only
position_concentration  -> portfolio_overlay
```

例子：

```text
MRVL:
  short_term: blocked，因为盘前/过热/急跌风险
  six_month: research_only 或 wait_entry，继续看 AI 网络/定制芯片订单兑现
  long_term: watch_only 或 focus_research，继续跟踪竞争力
```

### 1. 行业身份不匹配

示例：

```text
VIPS 因 KWEB 进入系统，但本质是中概电商/折扣零售平台。
它不能因为“中概互联网 + AI”标签进入 AI 核心推荐。
```

处理：

```text
降为 watch_only 或 research_only。
禁止进入 AI 核心推荐。
```

作用范围：

```text
scope = all
```

### 2. 过热风险

触发条件示例：

```text
1 年涨幅过大
短期涨幅过快
价格远离均线
成交拥挤
基本面兑现速度跟不上股价
```

处理：

```text
禁止追高。
只允许等回调支撑和订单确认。
```

作用范围：

```text
scope = short_term / six_month
```

### 3. 估值极端透支

触发条件示例：

```text
Forward PE 极端高
PS / EV/Sales 极端高
PEG 无法解释
收入增长和利润增长无法支撑估值
```

处理：

```text
从“买入”降为“观察买点”或“只观察”。
```

作用范围：

```text
scope = six_month
长期视角可保留 watch_only，但不能因为“长期好”而忽略当前估值风险。
```

### 4. 订单不确认

触发条件示例：

```text
只有新闻热度
没有财报收入
没有客户证据
没有订单或 backlog
没有指引上修
```

处理：

```text
不得进入重点研究。
```

作用范围：

```text
scope = six_month / research_only
```

### 5. 盘前风险闸门橙色或红色

盘前风险只使用现有 `premarket_gate` 作为单一来源。

```text
premarket_gate = NONE / LOW / HIGH / CRITICAL
```

它是 gate，不是三周期视角的新分项，也不是新评分。不得同时把同一组期货、利率、VIX、海外市场信号再写入其它评分，避免重复计数。

触发条件示例：

```text
期货大跌
利率急升
VIX 快速上行
AI 硬件链盘前集体下跌
韩国/日本/台股半导体先跌
重大宏观事件日
```

处理：

```text
不新开高 beta 仓位。
已有持仓按纪律线处理。
```

作用范围：

```text
scope = short_term
```

### 6. 持仓过重

持仓集中度不是 AI 推荐本体红旗，而是叠加在“我的持仓 / 今日决策台 / AI 组合方案差额”上的组合层提醒。

AI 推荐可以不知道用户真实持仓；真实持仓只能由用户确认交易后写入。

触发条件示例：

```text
单一股票超过总资产上限
同一产业链集中度过高
多个股票实际暴露在同一个风险因子上
```

处理：

```text
禁止继续加仓。
优先输出仓位复查。
```

作用范围：

```text
scope = portfolio_overlay
```

## 九、页面输出规则

AI 推荐页不能继续只显示一个排名。建议拆成四个区。

页面实现边界：

```text
这四个区应落在现有 AI 推荐/今日决策台的紧凑表格或折叠区里。
不得新开一个与今日决策台、AI 推荐、股票池竞争的第五入口。
列表优先用表格，保留 sticky 代码/名称首列和筛选。
卡片只用于单条 KPI 或风险摘要，不用大卡片堆列表。
```

### 1. 今晚/本周优先研究

回答：

```text
今天是否适合开仓？
如果适合，先看哪几只？
如果不适合，为什么？
```

展示字段：

```text
股票
短期动作
短期状态
参考价
买点条件
主要风险
盘前闸门影响
```

### 2. 未来半年主线候选

回答：

```text
当前产业阶段，哪些公司更受益？
```

展示字段：

```text
股票
所属分层
半年视角
订单/业绩证据
估值状态
回调状态
```

### 3. 一年以上科技成长核心池

回答：

```text
哪些公司长期值得跟踪？
```

展示字段：

```text
股票
长期视角
护城河
平台/生态

长期风险
```

### 4. 红旗拦截/只观察

回答：


展示字段：

```text
股票
旧系统分（仅解释，不参与本区排序）
拦截原因
```

### 5. 多市场展示要求

本规则当前示例以美股为主，但系统实际包含美股、A 股和港股。

```text
US：优先做完整试点，覆盖 AI 核心、AI 基建、软件、电力数据中心。
```

### 2. 推荐资格字段

```text
recommendation_eligibility:
  buyable
  research_only
  watch_only

```

`blocked` 不属于 eligibility，只属于 action（见 §7）。原因：一只票可以「资格没问题(eligibility=buyable)，但今天被盘前风险挡住(action=blocked)」——资格和当日动作是两件事，混在一个字段会互相污染。

拼写约定（避免代码踩坑）：剔除在 `eligibility` 写 `excluded`，在 `action` 写 `exclude`。`research_only / watch_only` 在两个字段同名，判断时必须带字段名。

## 十一、推荐生成逻辑

推荐系统应按以下顺序执行。

```text
1. 拉取和更新股票基础数据
2. 识别行业身份和科技成长多标签分层
3. 判断是否属于当前阶段受益方向
4. 收集订单、收入、客户、backlog、指引等证据状态
5. 展示短期/半年/长期三个周期视角，不直接合成新总分
6. 检查估值、过热、盘前风险等推荐本体红旗；其中盘前风险复用 premarket_gate
7. 生成动作：重点研究、等买点、只观察、红旗拦截、剔除
8. 保存 point-in-time 快照
9. 进入策略验证
10. 在今日决策台/真实持仓/AI 组合方案中再叠加持仓集中度提醒
```

伪代码：

```python
def build_recommendation(symbol, as_of):
    identity = classify_identity(symbol)
    if not identity.is_tech_growth:
        return exclude(symbol, "非科技成长方向")

    layers = classify_layers(symbol)  # primary_layer + secondary_layers
    eligibility = check_eligibility(symbol, layers)
    if eligibility in {"excluded", "watch_only"}:
        return watch_or_exclude(symbol, eligibility)

    evidence_status = collect_business_evidence_status(symbol, as_of)
    views = {
        "short_term": build_short_term_view(symbol, evidence_status, as_of),
        "six_month": build_six_month_view(symbol, layers, evidence_status, as_of),
        "long_term": build_long_term_view(symbol, evidence_status, as_of),
    }

    red_flags = detect_red_flags(symbol, views, evidence_status, as_of)
    action = decide_action(views, red_flags, eligibility)

    return Recommendation(symbol, layers, views, red_flags, action)
```

## 十二、策略验证规则

推荐规则不能只靠主观判断，必须持续验证。

验证口径必须复用现有 `stock_research/core/strategy_eval.py` 作为单一来源。

```text
不允许一个页面一套 SQL。
不允许一个脚本一套成熟样本口径。
不允许未去重、未过滤 NaN、未区分 PIT 的样本进入策略结论。
```

### 1. 必须保存历史快照

每次推荐必须保存当时的数据和结论，不能用后来的数据回填。

必须保存：

```text
run_id
generated_at
symbol
当时价格
当时 primary_layer
当时 secondary_layers
当时 ai_relevance_level
当时 recommendation_eligibility
当时 source_evidence
当时 classification_version
当时评分
当时动作
当时红旗
当时 red_flag_scope
当时市场环境
当时 market_phase_snapshot_id
```

说明：

```text
身份、分层、推荐资格也必须 PIT 保存。
不能用今天的 AI 身份去回测过去的推荐，否则会产生未来函数。
```

### 2. 验证口径

验证不能只看平均收益，必须看：

```text
中位数 alpha
胜率
最大亏损
最大回撤
Top 3 / Top 5 表现
是否靠极端赢家撑平均值
是否样本外仍有效
```

分层回测必须有最低样本量门槛：

```text
n < 20：只展示，不下结论
20 <= n < 50：弱参考，不允许升级策略
50 <= n < 100：可做初步判断
n >= 100：才可作为策略升级的重要依据
```

如果按“分层 × 周期 × 动作”切分后样本不足，应合并口径或延长观察期，不能强行下结论。

### 3. 不能用未来函数

策略验证只能用当时已知信息。

禁止：

```text
用后来补全的财报判断过去推荐
用后来涨幅倒推当时应该买
覆盖旧策略版本
只保留成功案例
```

### 4. 策略升级条件

从研究进入小仓试运行，至少需要：

```text
样本量达到最低门槛
中位数 alpha 转正
胜率明显高于 50%
最大亏损可控
不是由单个极端赢家贡献
样本外前瞻验证通过
```

## 十五、验收标准

### 1. 身份拦截验收

示例：

```text
VIPS 不得进入 AI 核心推荐。
仅 KWEB 来源股票不得直接强买。
```

### 2. 页面验收

AI 推荐页必须能看出：

```text
这只股票属于哪个层
短期能不能买
半年是不是主线
长期是否值得跟踪
为什么被拦截
```

### 3. 策略验证验收

必须能回答：

```text
新规则是否提高中位数 alpha
新规则是否提高胜率
新规则是否减少最大亏损
新规则是否只是靠少数大赢家
```

### 4. 真实持仓安全验收

任何推荐系统升级不得自动写真实持仓。

必须保持：

```text
AI 推荐是模型建议
AI 组合方案是目标组合
真实持仓只能由用户确认交易后写入
```

### 5. 四方审查清单

这份规则交付前，必须同时通过产品、测试、投研、工程四个视角。任何一项不通过，都不能宣布“可用于真钱决策”。

产品视角：

```text
用户能一眼看懂：这只票为什么出现、今天能不能动、主要风险是什么。
页面不裸露英文枚举，不让用户猜 research_only / buyable 的含义。
今日决策台只汇总，不新建股票池，不写 watchlist，不写真实持仓。
AI 推荐、AI 组合方案、已拉取股票池仍属于系统自动流；自选股仍属于用户手动流。
```

测试视角：

```text
有正向样本：NVDA/MRVL/AVGO/VST 等示例能按当前版本分到正确层。
有负向样本：VIPS/ETF-only/证据过期股票不能进入 focus_research。
有边界样本：同一股票既在 watchlist 又在系统池时，来源身份不混用。
有回归样本：打开页面、刷新推荐、生成组合，都不能自动改变真实持仓。
```

投研/风控视角：

```text
每只重点票必须能解释业务、证据、估值、买点、红旗和期限。
强势股不能因为热度高直接进入买入动作。
好公司但估值透支时，只能 wait_entry 或 watch_only。
短期风险不能抹掉长期跟踪价值；长期逻辑也不能覆盖今晚风险。
```

工程视角：

```text
单一来源：data_usability / quality_tag / premarket_gate / strategy_eval 不得平行再造。
PIT 保存：身份、证据、资格、动作、红旗、阶段快照必须可回放。
字段有来源：没有结构化来源的字段只能展示状态，不能参与真钱入口。
策略变更有版本：新权重/新阈值必须走验证，不得覆盖旧策略。
```

### 6. 回归测试最低清单

涉及代码落地时，至少要补以下检查。文档阶段不运行这些命令；实现阶段必须跑过并贴结果。

```text
1. schema 检查：
   recommendation_picks 是否有 eligibility/action/evidence_status/eligibility_migration_status
   system_universe 或 growth_layer 表是否有 primary_layer/secondary_layers_json

2. 来源边界检查：
   pool_membership.pool_type 只表示来源，不出现 ai_core/power_datacenter/excluded 等分层值
   watchlist 行数、holdings/real_holdings 行数在推荐生成前后不变

3. 推荐资格检查：
   eligibility 只允许 buyable/research_only/watch_only/excluded
   action 只允许 focus_research/wait_entry/watch_only/research_only/blocked/exclude
   evidence_status 只允许 confirmed/needs_review/candidate/stale/missing

4. 页面检查：
   AI 推荐页不显示裸英文枚举
   红旗拦截区显示“旧系统分仅解释，不参与排序”
   今日决策台不生成新池、不写自选、不写真实持仓

5. 策略验证检查：
   新规则进入 strategy_eval 的 PIT 样本
   n<20 的分层/动作样本只展示，不得下结论
   新口径不允许覆盖旧版本
```

推荐命令：

```bash

```

## 十六、一句话原则

系统以后不能再问：

```text
哪只股票分数最高？
```

而要先问：

```text
它是不是科技成长主线？
它处在哪个产业阶段？
订单和收入有没有确认？
估值有没有透支？
回调有没有支撑？
有没有红旗？
它适合今晚、本半年，还是长期？
```

只有这些问题都回答清楚，推荐才有资格进入用户决策。

## 17.3 eligibility 单一派生规则（后端算一次，前端只渲染）

在 [build_v2_recommendations.py](scripts/tools/build_v2_recommendations.py) 内按**有序**判定：

```text

4. 在分层内 且 evidence_status == confirmed                     → buyable
5. 在分层内 且 evidence_status ∈ {needs_review,candidate,stale,missing} → research_only
6. 其余                                                         → research_only
```

P0 的 `evidence_status` canonical 来源是 **`ai_theme_company_tags.evidence_status`**（库内取值 confirmed / needs_review / candidate / stale）。其中 `missing` 不是库里存的值，而是该 symbol 在 `ai_theme_company_tags` **无记录时的派生状态**，按"无证据"处理。§10.4 的 `order_evidence_status / revenue_confirmation_status / capex_beneficiary_status` 是后续细分维度，P0 不作为 buyable 判据。

`action` 派生规则（在 eligibility 之上再算一层，二者是不同字段）：

```text
eligibility == excluded                          → action = exclude
eligibility == watch_only                        → action = watch_only
eligibility == research_only                     → action = research_only
eligibility == buyable 且 命中 all-scope 红旗      → action = blocked，且必须解释全期限原因
eligibility == buyable 且 命中 short_term 红旗     → action = blocked，只挡今晚/本周
eligibility == buyable 且 买点未到(回调/确认不足)  → action = wait_entry
eligibility == buyable 且 买点成立                 → action = focus_research
```

注：`blocked` 只能由 short_term/all 范围红旗触发。short_term blocked 不改变 eligibility（资格仍是 buyable，只是今天被挡）；all-scope blocked 必须说明是否需要同步把 eligibility 降为 excluded 或 research_only。`position_concentration` 不允许在这里触发 blocked，只能进入 `portfolio_overlay_flags_json`。

# 十八、生产规则缺口与改进项（2026-06-10 实盘案例触发）

本章由 2026-06-10 的真实生产 run（`rec_20260610_231235_system_tech`）和一笔真实持仓（MRVL）暴露的问题倒推而来，是对**现有生产推荐规则**的改进清单。所有项严格遵守 §0 开发红线：**碰评分行为的必须先 shadow 验证 + 用户确认，不秘密改公式；能用 filter/tag 解决的优先 filter/tag。**

## 两个活样本（改进的依据）

```text
样本1 · 身份不符却被强烈推荐：
  唯品会 VIPS（中概电商/折扣零售，靠 KWEB 进系统）在 2026-06-10 run 里
  rank 48、rating=strong_buy、total_score=82.0，出现在美股 AI 推荐名单中。
  → 现有规则只看分数、不看身份，身份准入闸未接入生产。

样本2 · 过热票"靠运气"被漏掉：
  MRVL 一年 +318%、一月 +80%，极度过热；2026-06-10 未进推荐 run，
  但真实原因是当天 price_daily 的 momentum/fundamentals 字段缺失，被 data_usability gate 拦掉，
  并非系统识别了"过热"。数据齐全时，它可能在 blowoff 顶部附近被高分推荐。
  → 系统缺真正的过热护栏；data_usability gate 在"假装"处理了过热。

附带观察：当日 20 只美股 picks 全部 rating=strong_buy（阈值 ≥75）——评级可能在动量市里通胀。
```

## 18.1 ① 身份准入闸 — 立即做（filter/tag，不碰公式）

- **问题（大白话）**：系统推荐只看分数、不看公司是干什么的，导致卖打折衣服的唯品会进了 AI 强烈推荐。
- **动作**：落地 §17.3 的 `primary_layer` + `eligibility`：身份非科技成长 → `excluded`；仅 ETF 来源无第二证据 → `watch_only`；AI 推荐只展示 `buyable / research_only`。身份标签来自 §17.2 的手审白名单（`tech_growth_layers.py`），**不得写入 `pool_membership.pool_type`**。
- **性质**：纯过滤层，不动任何打分公式 → **可立即实施**。
- **落点**：见 §17.4 P0-A / P0-B（`tech_growth_layers.py` + `enrich_system_universe_v2.py` + `build_v2_recommendations.py`）。
- **验收**：用 2026-06-10 run 做前后对比，**VIPS 从 strong_buy → excluded（不出现在 AI 推荐）**；NVDA=ai_core 仍正常。

## 18.2 ② 过热护栏 — 需 shadow 验证后上线（碰评分行为）

- **问题（大白话）**：涨太猛的票（如 MRVL 年 +318%）风险极高，但系统没有专门识别"过热=危险"的逻辑，MRVL 这次是靠"数据缺失"被漏掉的，不是被看穿的。
- **动作**：**扩现有 price-action gate**（不新建公式），对"年涨幅极端 + 价格远离均线 + 抛物线形态"打 `overheated_flag`（对应 §8.2 过热红旗，scope=short_term / six_month）；命中后动作封顶为 `wait_entry / research_only`，不得给 strong_buy / focus_research。
- **性质**：改变系统给出的动作 → **必须先 shadow 验证**（用历史数据回算"这样拦的中位数 alpha/胜率/最大亏损是否改善"），通过 + 用户确认后才上线。阈值不得拍脑袋。
- **落点**：`build_v2_recommendations.py` 现有 price-action gate 扩展；阈值入 `calibrated_*` 配置，验证走 `strategy_eval.py`。
- **验收**：shadow 模式下，MRVL 这类（年 +318%、刚 blowoff）被标 overheated 且动作降级；回算指标不劣于现状。

## 18.3 ③ 过热票关闭"超跌反弹"加分 — 需 shadow 验证（碰评分行为）

- **问题（大白话）**：系统有条被验证有效的规律叫"超跌反弹"（reversal）——跌多了给加分、当作可捡便宜。但它在"先涨上天再崩盘"的票上会帮倒忙：MRVL 从 316 崩到 263，reversal 可能误判成"超跌可抄底"，而那其实是泡沫破裂、可能继续跌。
- **动作**：当 `overheated_flag` 触发时，**门控关闭该票的 reversal 加分**（不删因子、只在过热态下不计入）。
- **性质**：改变 reversal 因子的生效条件 → **必须先 shadow 验证**，确认不削弱 reversal 已验证的 alpha。
- **落点**：`build_v2_recommendations.py` 中 reversal 子权重计算处加 overheated 门控；验证走 `strategy_eval.py`。
- **验收**：shadow 下，过热票不再因 reversal 获得"抄底"加分；非过热票的 reversal alpha 不受影响。
- **依赖**：依赖 ② 的 `overheated_flag`，与 ② 同批验证。

## 18.4 ④ 区分"数据缺失排除" vs "风险排除" — 观测性，可立即加

- **问题（大白话）**：MRVL 是"因数据缺"被跳过的，但看起来像"系统聪明避开了过热票"，这是假象，会让人以为过热问题已解决（其实没有，要靠 ②）。
- **动作**：在 `recommendation_data_usability_audit.json` / 推荐审计里，对每只被排除的票**标明排除原因类别**：`excluded_by_data`（数据缺失）/ `excluded_by_risk`（过热/红旗）/ `excluded_by_rank`（未进 top-N）。
- **性质**：只记账、不改任何推荐逻辑 → **可立即实施**。
- **落点**：`build_v2_recommendations.py` 的 `_build_data_usability_audit` 扩展原因分类字段。
- **验收**：审计文件里 MRVL 当日标 `excluded_by_data`，而非被误读为风险规避。

## 18.5 ⑤ rating 阈值校准 — 需验证 + 用户确认

- **问题（大白话）**：今天 20 只美股 picks **全是 strong_buy**。当"强烈推荐"满天飞就失去意义（像全班都 95 分）。可能是普涨行情，也可能是 ≥75 这条线在动量市里太松。
- **动作**：复盘 `_rating` 的 ≥75/≥60 阈值在当前 regime 下是否过松；考虑按市场/regime 动态化或上调。
- **性质**：改评级标准 → **必须先验证 + 用户确认**，不擅自改（§0 红线1）。
- **落点**：`build_v2_recommendations.py` 的 `_rating`；校准与回算走 `strategy_eval.py`。
- **验收**：给出"当前阈值下 strong_buy 占比 vs 历史"的数据，由用户决定是否调整。
- **优先级**：最低，先观测。

# 十九、策略可信度体检 与「能不能照着买」（2026-06-11 投资经理视角）

## 一句话结论

```text
当前这套与其说是"能赚超额的策略"，不如说是"一个估值偏好的选股筛子"：
能帮你发现候选，但没有证据证明能跑赢；样本外它在亏、命中不到一半，
且高度集中在 AI 单一风险上。→ 适合当研究线索，不适合照着下单。
```

## 19.1 五视角诊断（真实数据，非印象）

实测打分构成（2026-06-11 US picks）：`总分 = 0.15×动量 + 0.50×估值 + 0.15×反转 + 0.20×数据完整度`。

```text
1) 量化/因子：~¾ 的分不是 alpha
   - 数据完整度占 20% 且每只都满分 100 → 白送 20 分；它不预测收益、且已另有 gate，重复计数。
   - 估值占 50% 主导，但这是 AI 成长/动量宇宙 → 系统性买"便宜的落后者"
     （ADSK/META/VST 动量仅 27-33 却被顶到前排）→ 极可能是样本外 alpha 转负之源。
   - 唯一被独立验证的 alpha 是 reversal，只占 15%；动量已被 IC 判失效。
2) 验证/统计（最致命）：样本内 +0.39%/命中83%，样本外 −0.8~−1.0%/命中 35-39%
   → 典型过拟合；按 §12.4 升级条件 FAIL。可信样本仅 2.5 周，正负都低置信，
     最严谨表述："目前没证明出任何 edge，全部当研究对待。"
3) 风控/组合：等权 top-10、无风险模型、不控因子暴露 → 10 只全压 AI capex
   = 单一主题 10 倍下注，"假分散"。dashboard 上 vs SPY 的超额大概率是 beta 不是 alpha。
4) 基本面：缺成长股最稳健的"盈利预期上修"信号；quality_score/ai_relevance 空架子。
5) 流程/定性：20/20 全 strong_buy → 评级零区分度；本质是筛子，却被叫"强烈推荐"，夸大了一个排名能给的东西。
```

## 19.2 「可以照着买」的 4 个硬条件（缺一不可）

```text
3. 每只有明确理由 + 退出纪律（不是只给名字）
4. 用户在 loop 里确认每一笔（真钱铁律，见 §15.4 / advisory-not-directive）
```
