# AI 与科技成长推荐规则

日期：2026-06-09

状态：产品规则草案，可作为后续推荐系统、股票池分层、页面展示和策略验证的对齐依据。

## 一、这份文档解决什么问题

本系统不能继续只做“分数高就推荐”。科技成长股，尤其 AI 相关股票，必须先判断身份和阶段，再看订单、估值、买点和风险。

本规则的目标是把 AI 推荐从“热点榜单”升级为：

```text
科技成长股票筛选 + 阶段识别 + 周期评分 + 风险拦截 + 可验证推荐
```

它解决以下问题：

1. 避免非 AI/非科技核心股票因为估值便宜、反转分高而误入 AI 推荐。
2. 避免 MRVL、AVGO、HPE、DELL 这类热门 AI 基建股在过热阶段被无脑追高。
3. 让 VST、GEV、VRT 这类数据中心电力/散热受益股能被正确识别，而不是被排除在 AI 主线之外。
4. 让 ADSK、INTU、CRM、ADBE 这类软件股进入“AI 集成/应用阶段”观察，而不是和芯片股混在同一个短线榜单里。
5. 让系统明确区分“今晚能不能买”“未来半年是否在主线”“长期是否值得跟踪”。

## 二、系统定位

StockAssistant 的推荐系统长期定位不是“AI 概念股推荐器”，而是“科技成长轮动系统”。

当前 AI 是最强主线，但未来科技成长主线可能在以下方向之间轮动：

```text
半导体
云计算
AI 算力
AI 网络
AI 应用软件
数据中心电力
液冷和基础设施
网络安全
工业自动化和机器人
医疗科技
互联网平台
金融科技
消费科技
```

因此系统必须保留科技成长大池，但在不同阶段识别不同受益链条。

## 三、当前产业阶段判断

截至 2026-06-09，本系统对 AI 产业阶段的判断为：

```text
AI 数据中心建设后半段
+ AI 推理/Agent/企业集成商业化前半段
```

这意味着：

1. 未来 1-8 周更看重市场风险、回调支撑、盘前环境和事件催化。
2. 未来 3-12 个月更看重订单、收入兑现、CapEx 受益和估值是否透支。
3. 未来 1-5 年更看重平台能力、数据、客户、生态、护城河和现金流。

系统推荐必须同时输出短期、中期、长期三个视角，而不是只给一个“强买”。

## 四、股票池分层

系统不再使用一个混合榜单直接推荐。所有股票必须先进入分层。

### 1. AI 核心池

含义：
- 直接提供 AI 算力、AI 网络、AI 芯片、AI 云平台或 AI 基础模型能力。

示例：

```text
NVDA
AVGO
MRVL
TSM
AMD
MU
CRDO
ALAB
MSFT
GOOGL
META
AMZN
```

推荐意义：
- 当前 AI 主线的核心观察池。
- 可以进入 AI 推荐和 AI 组合方案，但必须通过估值、订单、红旗和买点检查。

### 2. AI 基建池

含义：
- 受益于 AI 数据中心建设，但不一定直接提供 AI 芯片或模型。

示例：

```text
VRT
GEV
VST
HPE
DELL
SMCI
ETN
PWR
```

推荐意义：
- 当前阶段重要，但需要和 AI 核心池分开评分。
- 重点看订单、backlog、数据中心收入、毛利率、交付能力和估值透支。

### 3. 科技软件池

含义：
- AI 进入企业工作流后可能受益的软件、设计、财税、办公、CRM、安全等公司。

示例：

```text
ADSK
INTU
CRM
ADBE
NOW
PANW
CRWD
SNOW
DDOG
```

推荐意义：
- 更偏 6-18 个月和长期逻辑。
- 不应因为短期 AI 基建热度较弱就被忽略。
- 重点看 AI 功能是否带来付费率、留存率、ARPU、利润率或新产品增长。

### 4. 互联网平台池

含义：
- 拥有流量、数据、广告、搜索、电商、本地生活或内容分发能力的平台公司。

示例：

```text
GOOGL
META
AMZN
NFLX
UBER
SHOP
```

推荐意义：
- 重点看 AI 是否改善广告、搜索、推荐、云、客服、内容生产和成本结构。
- 不能只因为“互联网 + AI”标签就进入 AI 核心推荐。

### 5. 主题观察池

含义：
- 通过 ETF、新闻主题或宽泛行业线索发现，但尚未证明属于可买科技成长主线。

示例来源：

```text
KWEB
PAVE
ICLN
REMX
URA
LIT
```

推荐意义：
- 只能观察，不能直接推荐买入。
- ETF 持仓只能作为发现入口，不能作为“AI 相关性”的最终证据。

规则：

```text
仅 ETF 来源 + 无第二证据 = watch_only
宽泛行业主题 + 无订单/业务证据 = research_only
非科技/非成长/非主线 = excluded
```

## 五、推荐准入门槛

任何股票进入推荐前，必须先过准入门槛。

### 1. 行业身份门槛

系统必须明确回答：

```text
这家公司到底属于哪个行业？
它是不是科技成长方向？
它是不是当前阶段的受益者？
```

如果无法回答，不能进入推荐，只能进入观察。

### 2. AI/科技相关性门槛

AI 相关性不能只靠名字、新闻热词或 ETF 标签。

有效证据包括：

```text
财报中披露 AI 相关收入
管理层在财报电话会明确说明 AI 订单或客户需求
客户名单或供应链订单可验证
backlog 或指引上修来自 AI/数据中心需求
CapEx 直接流向该公司所在链条
产品被用于 AI 训练、推理、网络、数据中心、电力、散热或企业工作流
```

无效或弱证据包括：

```text
只在 ETF 中出现
只在新闻标题里出现 AI
只属于宽泛互联网或消费行业
只因为股价上涨被市场贴 AI 标签
```

### 3. 订单/业绩确认门槛

系统必须优先确认：

```text
收入是否增长
AI/数据中心收入是否增长
订单是否增长
backlog 是否增长
指引是否上修
毛利率是否稳定
客户是否真实
```

只讲故事、没有收入或订单确认的股票，不允许给“重点研究以上”的动作。

### 4. 估值不过度透支门槛

系统必须判断：

```text
PE / Forward PE 是否极端
PS / EV/Sales 是否极端
PEG 是否合理
估值是否高于同业太多
增长率能否支撑估值
```

好公司估值过满，也只能等买点，不能直接推荐。

### 5. 技术面和回调支撑门槛

系统必须判断：

```text
是否刚经历过急涨
是否处于过热区
是否出现放量下跌
是否跌破关键支撑
回调后是否开始企稳
相对强度是否仍优于行业
```

原则：

```text
不是越跌越买。
不是越涨越追。
回调必须有支撑，反弹必须有确认。
```

## 六、三周期评分

系统每只股票不再只有一个总分，而是至少输出三个分数。

```text
score_short_term     今晚/本周是否适合买
score_6m             未来半年是否处在主线
score_12m_plus       一年以上是否值得长期跟踪
```

### 1. 短期评分

适用问题：

```text
今晚能不能买？
本周能不能加仓？
急跌后能不能接？
```

建议权重：

```text
市场环境/盘前风险：25
技术面和回调支撑：25
事件催化：15
估值是否过热：15
订单/基本面确认：10
持仓和仓位风险：10
```

短期评分主要服务交易节奏，不代表长期价值。

### 2. 半年评分

适用问题：

```text
未来 3-12 个月是否还在主线？
当前产业阶段是否继续受益？
```

当前阶段建议权重：

```text
订单/客户/收入确认：30
AI 数据中心或 AI 集成受益程度：25
估值是否透支：20
回调是否有支撑：15
财务质量和竞争格局：10
```

半年评分是当前系统最重要的推荐评分。

### 3. 长期评分

适用问题：

```text
这家公司 1-5 年是否仍可能是科技成长核心资产？
```

建议权重：

```text
平台能力/护城河：25
客户和生态：20
数据/产品壁垒：15
长期利润率和现金流：15
行业空间：15
资产负债和资本开支风险：10
```

长期评分不能直接等于“今晚可以买”。

## 七、总分和动作定义

系统可以保留 100 分制，但动作必须重新定义。

```text
85 分以上：重点研究，可小仓试探
75-85：观察买点，等回调或确认
65-75：放池跟踪，不买
65 以下：剔除或仅观察
触发红旗：无论总分多高，禁止推荐
```

推荐动作必须使用以下枚举：

```text
focus_research      重点研究
wait_entry          等买点
watch_only          只观察
research_only       研究，不进真钱
blocked             红旗拦截
exclude             剔除
```

禁止只输出：

```text
强买
买入
无脑加仓
```

## 八、红旗一票否决

红旗优先级高于总分。触发红旗时，即使分数很高，也不能给买入建议。

### 1. 行业身份不匹配

示例：

```text
VIPS 因 KWEB 进入系统，但本质是中概电商/折扣零售平台。
它不能因为“中概互联网 + AI”标签进入 AI 核心推荐。
```

处理：

```text
降为 watch_only 或 research_only。
禁止进入 AI 核心推荐。
```

### 2. 过热风险

触发条件示例：

```text
1 年涨幅过大
短期涨幅过快
价格远离均线
成交拥挤
基本面兑现速度跟不上股价
```

处理：

```text
禁止追高。
只允许等回调支撑和订单确认。
```

### 3. 估值极端透支

触发条件示例：

```text
Forward PE 极端高
PS / EV/Sales 极端高
PEG 无法解释
收入增长和利润增长无法支撑估值
```

处理：

```text
从“买入”降为“观察买点”或“只观察”。
```

### 4. 订单不确认

触发条件示例：

```text
只有新闻热度
没有财报收入
没有客户证据
没有订单或 backlog
没有指引上修
```

处理：

```text
不得进入重点研究。
```

### 5. 盘前风险闸门橙色或红色

触发条件示例：

```text
期货大跌
利率急升
VIX 快速上行
AI 硬件链盘前集体下跌
韩国/日本/台股半导体先跌
重大宏观事件日
```

处理：

```text
不新开高 beta 仓位。
已有持仓按纪律线处理。
```

### 6. 持仓过重

触发条件示例：

```text
单一股票超过总资产上限
同一产业链集中度过高
多个股票实际暴露在同一个风险因子上
```

处理：

```text
禁止继续加仓。
优先输出仓位复查。
```

## 九、页面输出规则

AI 推荐页不能继续只显示一个排名。建议拆成四个区。

### 1. 今晚/本周优先研究

回答：

```text
今天是否适合开仓？
如果适合，先看哪几只？
如果不适合，为什么？
```

展示字段：

```text

```

### 2. 未来半年主线候选

回答：

```text

```

展示字段：

```text

```

### 3. 一年以上科技成长核心池

回答：

```text

```

展示字段：

```text
股票
```

### 4. 红旗拦截/只观察


## 十一、推荐生成逻辑

推荐系统应按以下顺序执行。

```text
1. 拉取和更新股票基础数据
2. 识别行业身份和科技成长分层
3. 判断是否属于当前阶段受益方向
4. 收集订单、收入、客户、backlog、指引等证据
5. 计算短期/半年/长期三个周期评分
6. 检查估值、过热、盘前风险、持仓集中度等红旗
7. 生成动作：重点研究、等买点、只观察、红旗拦截、剔除
8. 保存 point-in-time 快照
9. 进入策略验证
```

伪代码：

```python
def build_recommendation(symbol, as_of):
    identity = classify_identity(symbol)
    if not identity.is_tech_growth:
        return blocked(symbol, "非科技成长方向")

    layer = classify_layer(symbol)
    eligibility = check_eligibility(symbol, layer)
    if eligibility in {"excluded", "watch_only"}:
        return watch_or_exclude(symbol, eligibility)

    evidence = collect_business_evidence(symbol, as_of)
    scores = {
        "short_term": score_short_term(symbol, evidence, as_of),
        "six_month": score_six_month(symbol, layer, evidence, as_of),
        "long_term": score_long_term(symbol, evidence, as_of),
    }

    red_flags = detect_red_flags(symbol, scores, evidence, as_of)
    action = decide_action(scores, red_flags, eligibility)

    return Recommendation(symbol, layer, scores, red_flags, action)
```

## 十二、策略验证规则

推荐规则不能只靠主观判断，必须持续验证。

### 1. 必须保存历史快照

每次推荐必须保存当时的数据和结论，不能用后来的数据回填。

必须保存：

```text
run_id
generated_at
symbol
当时价格
当时分层
当时评分
当时动作
当时红旗
当时市场环境
```

### 2. 验证口径

验证不能只看平均收益，必须看：

```text
中位数 alpha
胜率
最大亏损
最大回撤
Top 3 / Top 5 表现
是否靠极端赢家撑平均值
是否样本外仍有效
```

### 3. 不能用未来函数

策略验证只能用当时已知信息。

禁止：

```text
用后来补全的财报判断过去推荐
用后来涨幅倒推当时应该买
覆盖旧策略版本
只保留成功案例
```

### 4. 策略升级条件

从研究进入小仓试运行，至少需要：

```text
样本量达到最低门槛
中位数 alpha 转正
胜率明显高于 50%
最大亏损可控
不是由单个极端赢家贡献
样本外前瞻验证通过
```

## 十三、和现有系统的关系

这份规则不是推翻整个系统。

保留：

```text
行情数据
估值数据
涨跌幅和动量数据
SEC / 13F / 新闻 / 财报事件
盘前预警
策略验证
持仓纪律
推荐历史样本
```

需要升级：

```text
股票池身份分层
ETF 来源降级规则
推荐资格字段
订单/收入证据评分
短期/半年/长期三周期评分
红旗一票否决
页面动作表达
策略验证口径
```

必须推翻：

```text
分数高就推荐
ETF 来源直接进 AI 推荐
把短期交易分、半年主线分、长期质量分混成一个解释不清的总分
```

## 十四、落地优先级

### P0：先修推荐准入

目标：

```text
避免 VIPS 这类身份不匹配股票进入 AI 推荐。
```

动作：

```text
增加 tech_growth_layer
增加 recommendation_eligibility
ETF-only 默认 watch_only
非科技成长默认 excluded
AI 推荐只允许 buyable / research_only 中符合条件的股票展示
```

### P1：增加三周期评分

目标：

```text
同一只股票明确区分短期、半年、长期价值。
```

动作：

```text
增加 score_short_term
增加 score_6m
增加 score_12m_plus
页面按周期展示
```

### P2：红旗一票否决

目标：

```text
高分但高风险股票不能直接推荐。
```

动作：

```text
过热红旗
估值红旗
订单不确认红旗
盘前风险红旗
持仓集中度红旗
```

### P3：策略验证升级

目标：

```text
验证新规则是否真的比旧规则好。
```

动作：

```text
按分层回测
按周期回测
按动作回测
看中位数、胜率、最大亏损和样本外表现
```

## 十五、验收标准

### 1. 身份拦截验收

示例：

```text
VIPS 不得进入 AI 核心推荐。
仅 KWEB 来源股票不得直接强买。
```

### 2. 页面验收

AI 推荐页必须能看出：

```text
这只股票属于哪个层
短期能不能买
半年是不是主线
长期是否值得跟踪
为什么被拦截
```

### 3. 策略验证验收

必须能回答：

```text
新规则是否提高中位数 alpha
新规则是否提高胜率
新规则是否减少最大亏损
新规则是否只是靠少数大赢家
```

### 4. 真实持仓安全验收

任何推荐系统升级不得自动写真实持仓。

必须保持：

```text
AI 推荐是模型建议
AI 组合方案是目标组合
真实持仓只能由用户确认交易后写入
```

## 十六、一句话原则

系统以后不能再问：

```text
哪只股票分数最高？
```

而要先问：

```text
它是不是科技成长主线？
它处在哪个产业阶段？
订单和收入有没有确认？
估值有没有透支？
回调有没有支撑？
有没有红旗？
它适合今晚、本半年，还是长期？
```

只有这些问题都回答清楚，推荐才有资格进入用户决策。
