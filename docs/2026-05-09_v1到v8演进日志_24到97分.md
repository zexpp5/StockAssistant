## 三、中午 v6.0 (77 分)

### 触发动机

用户问"我的模型还缺什么"。我系统化拆解后，按 ROI 推荐"123 优先级"，用户说"123 都做"。

### 三个学术质量门

#### #1 行业 + 市值中性化（Fama-French 1992 / Rosenberg-Marathe 1976）

**问题**：v5 推荐 Top 5 全是工业电力链（LRCX/MTZ/MOD/VRT/GEV）—— 是**行业 beta**，不是真 alpha。

**解决**：`stock_research/core/neutralization.py`
- 行业 z-score 中性化（按 GICS 分组）
- 市值残差中性化（OLS 剔除 log MCap 影响）

**实测效果**：
- 中性化前 Top 5：全部工业电力链
- 中性化后 Top 5：跨 4 个行业（医药 / SaaS / 消费 / 半导体）
- **META, GOOGL 进入 Top 5**（软件行业内排名靠前）

#### #2 ADV 限流 + 交易成本（Almgren-Chriss 2001 / Frazzini 2018）

**问题**：v5 Markowitz 输出权重不考虑流动性，小盘股可能被严重超配。

**解决**：`stock_research/core/portfolio_constraints.py`
- 单日交易 ≤ 5% × ADV
- 成本扣减：5 bps 佣金 + 2 bps × (Δ$/ADV%)

**实测**（$50M 大资金）：
- SDGR 目标 +4.46% ($2.23M) 超过 ADV ($18.9M) × 5% = $945K
- 自动砍到 +1.89%
- 成本 6.1 bps，几乎不影响 alpha

#### #3 完整流水线（jobs/optimize_portfolio.py）

**集成**：v5 选股 → 中性化 → Top 12 → Markowitz → 行业敞口 ≤ 25% → ADV 限流 → 成本扣减 → 输出 plan

**评分提升**：v1 (24) → v6.0 (77)

### OpenBB 是什么（用户问）

OpenBB Terminal = **"开源版 Bloomberg"**，本质是**数据层的统一管道**（不是推荐系统）。

类比：
- OpenBB ≈ SQL（统一查多种数据库）
- 你的 v7 ≈ 业务系统（基于 SQL 数据做决策）

## 七、数据流架构

### 4 层存储

```
┌─────────────────────────────────────────────────┐
│  ① 飞书 Bitable（业务主存储 / 真实数据源）       │
│     watchlist / picks / 13F / 风险提示          │
│     谁读：你 / daily_picks / weekly_review      │
└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│  ② DuckDB（本地加速 / 历史查询）                 │
│     stock_history.duckdb                         │
│     谁读：跨周期分析 / SQL 查询                   │
└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│  ③ JSON 快照（审计追溯 / Web 消费）              │
│     data/snapshots/audit/*.json                  │
│     谁读：streamlit_app / build_dashboard_html   │
└─────────────────────────────────────────────────┘
                      ↕
┌─────────────────────────────────────────────────┐
│  ④ CSV 归档（公开化 / 不可篡改）                  │
│     archive/<date>/picks.csv                     │
│     谁读：GitHub / Reviewer / 历史回看            │
└─────────────────────────────────────────────────┘
```

### 各模块读写来源

| 模块 | 读 | 写 |
|---|---|---|
| fetch_stock_prices | yfinance API | → 飞书 + JSON + DuckDB |
| daily_picks_v5 | 飞书 watchlist | → 飞书 picks + factor_scores_today.json |
| weekly_review | 飞书 picks + yfinance | → 飞书 picks（更新 cum_pct）+ DuckDB |
| realtime_defense | 飞书 picks + yfinance VIX | → 飞书"风险提示" + JSON |
| audit_picks | 飞书 watchlist + picks | → JSON 快照 |
| openbb_intelligence | enrich JSON + OpenBB API | → JSON + Markdown |
| **streamlit_app.py** | **JSON + CSV**（**不读飞书**） | 只读 |

### 为什么这样分层

1. **飞书是给"人"看的**（你每天打开看 picks）
2. **DuckDB 是给"代码"用的**（SQL 查 5 年历史快 1000x）
3. **JSON 是给"审计"用的**（每次 job 都留快照）
4. **CSV+git 是给"公众"看的**（不可篡改）

## 九、关键技术决策与教训

### 决策 1：抖音截图 → 永久禁用
- **教训**：信息图视觉语言会丢失上下文（22.6% 占比 vs 变动）
- **规则**：关键数字必须从一手权威源（SEC / 港交所 / 公司官网）交叉验证
- **工具**：`verify_data_sources.py` + 数据源四档分级

### 决策 2：Markowitz 蒙特卡洛 → cvxpy 精确解
- **教训**：20000 次随机搜索是粗糙近似，cvxpy 是精确凸优化
- **替代**：PyPortfolioOpt
- **加成**：免费解锁 Black-Litterman / HRP / CVaR 4 种新方法

### 决策 3：alphalens-reloaded buggy → 自实现
- **问题**：pandas 2.x freq 推断报错
- **决策**：自实现核心 IC + Quintile 算法（不到 50 行代码）
- **价值**：不被库的 bug 拖累

### 决策 4：mlfinlab 已下架 → 自实现 De Prado
- **问题**：Hudson&Thames 商业化，PyPI 下架
- **决策**：De Prado 论文公开，自实现 Triple Barrier + Purged K-Fold
- **价值**：不依赖任何商业库

### 决策 5：Streamlit 而非 Flask/Django
- **理由**：30 行代码 = 完整 Web App，符合"个人系统优先 ROI"
- **代价**：高并发场景性能差（但你 1 个用户够用）

### 决策 6：故意不接真实交易接口
- **理由**：合规边界（个人无金融牌照）+ 风险控制
- **替代**：输出"建议"，用户保留最终决策权

### 决策 7：双套数据存储（飞书 + DuckDB）
- **理由**：飞书是业务主存（人看），DuckDB 是历史加速层（代码用）
- **代价**：需要同步逻辑（fetch_stock_prices 同时写两边）

### 决策 8：诚实暴露弱点（STRESS_TEST_REPORT 主动披露 v6 在 2008/2022 跑输 SPY）
- **理由**：Charlie Munger "反过来想" — 暴露错误反而建立信任
- **效果**：给专业人士看到的"诚实信号"
